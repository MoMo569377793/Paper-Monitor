# 论文监控程序使用手册

## 1. 这套程序能做什么

这是一套本地运行的论文监控工具，当前默认监控两个方向：

- `有限元分析 Matrix-Free 算法优化`
- `AI 算子加速`

它会自动完成这些事情：

- 从 `arXiv`、`DBLP` 抓取论文
- 去重、主题匹配、相关性打分
- 下载 PDF，抽取全文文本
- 用一个或多个 LLM 生成逐篇中文总结
- 生成日报、周报、论文库总览
- 为每篇论文单独导出 Markdown / HTML / JSON 报告

运行时会显示实时进度条，包括：

- 抓取进度
- 单篇增强进度
- 日报 / 周报 / 总览生成进度

## 2. 使用前提

下面所有命令都默认在项目根目录执行。

也就是说，你应该先进入这个项目目录，再执行后面的命令。

推荐环境：

- `Python 3.11+`
- Linux / macOS / WSL
- 推荐安装：
  - `pdftotext`
  - `pdfinfo`

如果系统里有 `pdftotext / pdfinfo`，全文抽取会更稳定。

## 3. 配置文件说明

当前项目里你最常用的配置有 4 份：

- `config/config-poe.json`
  - 主配置
  - 当前模型：`gemini-3.1-pro`
- `config/config-ikun.json`
  - 第二模型配置
  - 当前模型：`gpt-5.4`
- `config/config-SAURLAX.json`
  - 可选旧配置
  - 当前模型：`deepseek-chat`
- `config/config.example.json`
  - 纯模板
  - 不直接连接任何真实后端

默认建议：

- 主模型用 `config/config-poe.json`
- 第二模型用 `config/config-ikun.json`

## 4. 先配置 API Key

推荐在项目根目录创建一个本地环境文件：

```bash
cat > .codex.env <<'EOF'
export POE_API_KEY='你的 Poe API Key'
export IKUNCODE_API_KEY='你的 Ikun API Key'
export SAURLAX_API_KEY='你的 SAURLAX API Key'
EOF
```

每次运行前先加载：

```bash
source .codex.env
```

注意：

- 不要把真实密钥直接写进配置文件里的 `api_key_env`
- `api_key_env` 里应该写环境变量名，不是密钥本身

## 4.1 怎么控制不同步骤的 reasoning_effort / thinking_level

主项目现在已经支持在这些步骤里读取推理强度配置：

- 单篇论文总结
- 全文分块摘要
- 全文聚合总结
- 领域总结
- 日报 / 周报 / 总览里的 LLM 聚合总结

最常用的配置键有 5 个：

- `llm.model_reasoning_effort`
  - 这个模型的默认推理强度
- `llm.model_thinking_level`
  - 主要给 `Poe + Gemini` 这类后端使用
- `llm.extra_body`
  - 原样透传给 `chat/completions`
- `llm.reasoning_by_task`
  - 按任务单独覆盖推理强度
- `llm.thinking_level_by_task`
  - 按任务单独覆盖 thinking level

当前支持的任务名：

- `paper_summary`
  - 单篇论文最终总结
- `paper_chunk`
  - 全文分块读取时的分块备忘录
- `paper_reduce`
  - 把全文分块结果聚合成最终单篇总结
- `topic_digest`
  - 领域聚合总结，用在日报 / 周报 / 总览
- `json_repair`
  - 模型输出 JSON 不规范时的修复请求

推荐写法 1：给整个模型设置统一强度

```json
"llm": {
  "model_reasoning_effort": "xhigh"
}
```

推荐写法 2：按任务分别控制

```json
"llm": {
  "model_reasoning_effort": "low",
  "reasoning_by_task": {
    "paper_summary": "high",
    "paper_chunk": "low",
    "paper_reduce": "high",
    "topic_digest": "xhigh",
    "json_repair": "low"
  }
}
```

如果你使用 `Poe + Gemini`，更推荐写法是：

```json
"llm": {
  "model": "gemini-3.1-pro",
  "model_thinking_level": "high"
}
```

或者：

```json
"llm": {
  "model": "gemini-3.1-pro",
  "extra_body": {
    "thinking_level": "high"
  }
}
```

说明：

- `Ikun / gpt-5.4` 更适合用 `model_reasoning_effort`
- `Poe / Gemini` 更适合用 `model_thinking_level` 或 `extra_body.thinking_level`
- 如果两边都配置了，任务级覆盖优先于模型级默认值

## 4.2 如何使用 `chat_with_config.py` 和指定模型直接对话

项目根目录下有一个独立脚本：

- `chat_with_config.py`

它的作用是：

- 读取你指定的 `config/*.json`
- 使用这份配置里的 LLM 直接对话
- 方便你单独测试某个 API 是否可用
- 方便你调试 `reasoning_effort / thinking_level`

它不会做这些事：

- 不抓论文
- 不下载 PDF
- 不写数据库
- 不生成日报 / 周报

也就是说，它只是一个“直接和模型聊天”的调试脚本。

### 最常用的两种用法

先加载环境变量：

```bash
source .codex.env
```

和 `poe` 模型对话：

```bash
python3 chat_with_config.py --config config/config-poe.json
```

和 `ikun` 模型对话：

```bash
python3 chat_with_config.py --config config/config-ikun.json
```

### 单次提问

如果你不想进入交互模式，可以直接一次性提问：

```bash
python3 chat_with_config.py --config config/config-poe.json --message "请用中文解释一下 Tensor Core 和普通 CUDA Core 的区别"
```

### 多行输入

如果你想一次发多行内容，推荐这样写：

```bash
python3 chat_with_config.py \
  --config config/config-poe.json \
  --message $'请帮我做三件事：\n1. 总结这篇论文\n2. 提取关键方法\n3. 给出局限性'
```

也可以从文件读取：

```bash
python3 chat_with_config.py \
  --config config/config-poe.json \
  --message "$(cat prompt.txt)"
```

### 不使用系统提示词

如果你只想把自己的 `message` 原样发给模型，可以这样：

```bash
python3 chat_with_config.py --config config/config-poe.json --system "" --message "你的问题"
```

### 查看实际 payload

如果你想检查脚本到底往后端发了什么，可以加：

```bash
--verbose
```

例如：

```bash
python3 chat_with_config.py --config config/config-ikun.json --message "你好" --verbose
```

这会打印：

- 实际请求的 endpoint
- 实际 payload 的关键字段

这对检查下面这些问题很有帮助：

- 当前到底用了哪个模型
- 有没有带上 `reasoning_effort`
- Poe / Gemini 有没有带上 `extra_body.thinking_level`

### 临时覆盖推理强度

如果你不想改配置文件，也可以直接在命令行临时覆盖：

```bash
python3 chat_with_config.py \
  --config config/config-ikun.json \
  --reasoning-effort xhigh \
  --message "请详细分析这段方法设计"
```

### 交互模式里的快捷命令

- `/exit` 或 `/quit`
  - 退出
- `/clear`
  - 清空当前对话历史

### 什么时候适合先用这个脚本

推荐你在下面这些场景先用 `chat_with_config.py`：

- 新接入一个 API URL，先测试能不能正常回答
- 想验证 `reasoning_effort / thinking_level` 是否真的生效
- 想先测试 prompt，再决定要不要放进主流程
- 想确认某个模型适不适合你的论文总结任务

## 5. 第一次使用：抓两个领域各 100 篇

这是最推荐的一套首次初始化流程。

### 第 1 步：抓取论文

```bash
python3 main.py --config config/config-poe.json fetch --source arxiv --source dblp --recent-limit 100 --page-size 50
```

这条命令的含义是：

- 从 `arXiv` 和 `DBLP` 抓取论文
- 对每个领域分别保留最近的 `100` 篇候选论文
- 然后写入数据库

这里的 `--recent-limit 100` 是：

- 每个领域 100 篇
- 不是总共 100 篇

因为当前有两个领域，所以首次抓取后，数据库里通常会有大约 `200` 篇论文。

### 第 2 步：生成逐篇 LLM 总结

```bash
python3 main.py --config config/config-poe.json enrich --limit 200 --with-llm --extra-config config/config-ikun.json --workers 1
```

这条命令会：

- 下载 PDF
- 抽取全文
- 自动检测模型是否支持直接读 PDF
- 如果支持，优先直接读 PDF
- 如果不支持，自动回退到“全文文本分块总结”
- 同时使用两个模型生成逐篇总结：
  - `poe`
  - `ikun`

说明：

- `--limit 200` 是总处理上限
- 因为前一步通常最多抓到两个领域合计约 `200` 篇，所以这里用 `200`
- `--workers 1` 最稳，适合首次跑全量

如果你非常在意“严格按领域各 100 篇”地分别增强，可以分两次跑：

```bash
python3 main.py --config config/config-poe.json enrich --limit 100 --topic matrix_free_fem --with-llm --extra-config config/config-ikun.json --workers 1
python3 main.py --config config/config-poe.json enrich --limit 100 --topic ai_operator_acceleration --with-llm --extra-config config/config-ikun.json --workers 1
```

### 第 3 步：生成论文库总览

```bash
python3 main.py --config config/config-poe.json catalog-report --extra-config config/config-ikun.json
```

这一步会生成一个“全库总览”，你可以直接看到：

- 每个领域目前有哪些论文
- 每篇论文的基础信息
- `poe` 的逐篇总结
- `ikun` 的逐篇总结

### 第 4 步：生成周报

```bash
python3 main.py --config config/config-poe.json report --type weekly --days 7 --with-llm --extra-config config/config-ikun.json
```

这一步会生成两个领域的 7 天周报，并让两个模型分别给出领域级总结。

## 6. 以后每天怎么跑

日常最推荐的是这一条一键命令：

```bash
python3 main.py --config config/config-poe.json run-once --type daily --source arxiv --source dblp --page-size 20 --since-last-run --enrich --enrich-limit 20 --with-llm --extra-config config/config-ikun.json --workers 1
```

这条命令会依次执行：

- 只抓自上次成功抓取后新增的论文
- 只增强这批新增论文
- 用 `poe + ikun` 生成逐篇总结
- 生成当天日报
- 导出对应的单篇论文报告

如果你想拆开执行，也可以这样：

```bash
python3 main.py --config config/config-poe.json fetch --source arxiv --source dblp --page-size 20 --since-last-run
python3 main.py --config config/config-poe.json enrich --limit 20 --with-llm --extra-config config/config-ikun.json --workers 1 --since-last-run
python3 main.py --config config/config-poe.json report --type daily --with-llm --extra-config config/config-ikun.json
```

## 8. 单篇论文总结是不是全文总结

默认情况下，程序会尽量做“基于全文”的总结，而不是只看摘要。

顺序是：

1. 优先尝试让模型直接读取 PDF
2. 如果后端不支持 PDF，就回退到：
   - `PDF -> 本地抽全文文本 -> 分块总结 -> 聚合`
3. 如果连全文都拿不到，才退到：
   - `标题 + 摘要 + 元数据`

如果你希望尽量生成全文总结，注意两点：

- 不要使用 `--skip-pdf`
- 尽量保证本地能成功下载 PDF

## 9. 怎么看某篇论文到底是不是“读全文”生成的

现在每个模型的逐篇总结都会明确标注输入依据。

你会看到三种之一：

- `已直接读取 PDF`
- `已读取完整全文`
- `仅基于摘要/元数据`

如果你想单独看某一篇论文，直接生成单篇报告：

```bash
python3 main.py --config config/config-poe.json paper-report --paper-id 133 --extra-config config/config-ikun.json
```

然后查看这篇论文在 `reports/papers/` 里的报告即可。

## 10. 当前数据库里有哪些论文、每个模型写了什么

如果你想看整个数据库的总览，用：

```bash
python3 main.py --config config/config-poe.json catalog-report --extra-config config/config-ikun.json
```

这条命令会显示：

- 每个领域的论文列表
- 每篇论文的默认总结
- `poe` 的逐篇总结
- `ikun` 的逐篇总结

现在总览、日报、周报、单篇报告默认只显示“当前命令启用的模型”。

例如你运行的是：

```bash
python3 main.py --config config/config-poe.json catalog-report --extra-config config/config-ikun.json
```

那么报告里只会显示：

- `poe`
- `ikun`

不会再自动把以前历史上用过的其他模型带出来。

## 11. 只想用一个模型，或者同时用两个模型

### 只用 `poe`

```bash
python3 main.py --config config/config-poe.json enrich --limit 20 --with-llm --workers 1
```

### 只用 `ikun`

```bash
python3 main.py --config config/config-ikun.json enrich --limit 20 --with-llm --workers 1
```

### 同时用 `poe + ikun`

```bash
python3 main.py --config config/config-poe.json enrich --limit 20 --with-llm --extra-config config/config-ikun.json --workers 1
```

如果你把顺序反过来：

```bash
python3 main.py --config config/config-ikun.json enrich --limit 20 --with-llm --extra-config config/config-poe.json --workers 1
```

那么：

- 两个模型都会生成逐篇总结
- 但“默认总结”会优先采用 `ikun`

```bash
python3 main.py --config config/config-poe.json enrich --paper-id 123 --with-llm --extra-config config/config-ikun.json --workers 1
```

## 13. 修改两个领域的关键词，应该改哪里

改你主配置文件里的 `topics` 段。

你现在主用的是：

- `config/config-poe.json`

也就是说，平时要改关键词，优先改这份配置。

重点看这些字段：

- `source_queries`
  - 真正发给 `arXiv`、`DBLP` 的检索词
- `must_match_groups`
  - 必须命中的关键词组
- `positive_keywords`
  - 加分关键词
- `exclude_keywords`
  - 排除词
- `arxiv_categories`
  - 限制 arXiv 分类
- `priority_arxiv_categories`
  - 提高某些 arXiv 分类优先级
- `dblp_venue_keywords`
  - 限制 DBLP 会议 / 期刊线索
- `priority_venue_keywords`
  - 提高某些会议 / 期刊优先级
- `threshold`
  - 相关性阈值

当前两个 topic id 是：

- `matrix_free_fem`
- `ai_operator_acceleration`

如果你只是想改“抓取时搜什么”，优先改：

- `source_queries`

如果你想改“抓回来以后哪些算真正相关”，优先改：

- `must_match_groups`
- `positive_keywords`
- `exclude_keywords`
- `priority_arxiv_categories`
- `priority_venue_keywords`

## 14. 如何接入一个新的 LLM URL

通常不需要改代码，只需要新增一份配置文件。

最简单的方式：

1. 复制模板
2. 修改新后端的 LLM 信息
3. 配环境变量

例如：

```bash
cp config/config.example.json config/config.new.json
```

然后修改 `config/config.new.json` 里的 `llm` 段，至少改这几个字段：

- `variant_id`
  - 必须唯一
- `label`
  - 报告里显示的名字
- `provider`
  - `openai_compatible` 或 `openai_responses`
- `base_url`
  - 例如 `https://your-api.example.com/v1`
- `api_key_env`
  - 例如 `NEW_LLM_API_KEY`
- `model`
  - 例如 `gpt-5.4`

然后在 `.codex.env` 里加上：

```bash
export NEW_LLM_API_KEY='你的新 API Key'
```

## 15. 如何检测新 URL 是否支持直接读 PDF

程序会自动检测，不需要你手写检测代码。

最简单的验证方式：

1. 先抓一篇论文并下载到本地 PDF
2. 用新模型跑一次 `enrich`
3. 导出单篇报告
4. 看报告里的 `输入依据`

例如：

```bash
python3 main.py --config config/config.new.json enrich --limit 1 --with-llm --workers 1
python3 main.py --config config/config.new.json paper-report --paper-id 1
```

判断方法：

- 如果显示 `已直接读取 PDF`
  - 说明这个后端支持 PDF 输入
- 如果显示 `已读取完整全文`
  - 说明程序自动回退到了全文文本模式
- 如果显示 `仅基于摘要/元数据`
  - 说明这次没有成功拿到可用全文

## 16. prompts 提示词在什么情况下使用

项目里的提示词文件都在：

- `prompts/paper_summary_system.txt`
- `prompts/paper_summary_user.txt`
- `prompts/paper_chunk_system.txt`
- `prompts/paper_chunk_user.txt`
- `prompts/paper_reduce_system.txt`
- `prompts/paper_reduce_user.txt`
- `prompts/topic_digest_system.txt`
- `prompts/topic_digest_user.txt`

### 16.1 `paper_summary_*` 在什么时候使用

这组提示词用于“单篇论文最终总结”。

它会在两种情况下使用：

1. 模型支持直接读取 PDF
   - 这时程序会把 PDF 直接交给模型
   - 使用的就是 `paper_summary_system.txt` 和 `paper_summary_user.txt`
2. 没有 PDF / 没有全文，只能基于摘要和元数据总结
   - 这时仍然使用 `paper_summary_system.txt` 和 `paper_summary_user.txt`

也就是说：

- 如果是“直接输入 PDF 给模型总结”，使用的就是 `paper_summary_*`

### 16.2 `paper_chunk_*` 在什么时候使用

这组提示词用于“全文文本分块分析”。

只有在下面这种情况下才会使用：

- 模型不支持直接读取 PDF
- 但本地已经成功抽取出了全文文本

这时程序会先把全文切成多块，然后每一块都使用：

- `paper_chunk_system.txt`
- `paper_chunk_user.txt`

生成分块备忘录。

### 16.3 `paper_reduce_*` 在什么时候使用

这组提示词用于“把多个分块备忘录聚合成单篇最终总结”。

它总是和 `paper_chunk_*` 配套使用。

也就是说：

- `paper_chunk_*` 负责逐块读全文
- `paper_reduce_*` 负责把全文分块结果合成最终单篇总结

### 16.4 `topic_digest_*` 在什么时候使用

这组提示词用于“领域级总结”，也就是：

- 日报里的领域总结
- 周报里的领域总结
- 论文库总览里的领域级概览

这一步不会重新读 PDF。

它读取的是：

- 当前模型自己生成的逐篇总结
- 或者回退到默认总结

然后再用：

- `topic_digest_system.txt`
- `topic_digest_user.txt`

生成某个领域的聚合总结。

### 16.5 如果我想改 prompt，应该改哪个文件

如果你想改：

- “每篇论文怎么总结”
  - 改 `paper_summary_*`
- “全文回退模式下，每个分块提取什么信息”
  - 改 `paper_chunk_*`
- “全文回退模式下，最后怎么把整篇论文合成一条总结”
  - 改 `paper_reduce_*`
- “日报 / 周报 / 总览里的领域总结”
  - 改 `topic_digest_*`

### 16.6 当前这版 prompt 已经针对你的需求优化了什么

当前 prompt 已经更强调下面这些信息：

- 对 `Matrix-Free FEM`
  - 是否是实现优化而不是纯数学分析
  - 是否涉及 `partial assembly`、`sum factorization`、`operator application`
  - 是否涉及 `GPU`、`SIMD`、`performance portability`
  - 是否使用 `libCEED`、`MFEM`、`deal.II`
- 对 `AI 算子加速`
  - 是否涉及 `GEMM`、`matmul`、`attention`、`kernel fusion`
  - 是否涉及 `Tensor Core`、`ARM SVE/SME`
  - 是否涉及 `CUTLASS`、`Triton`、`MLIR`
  - 是否给出明确性能结果、平台信息、基线对比

### 16.7 修改 prompt 以后什么时候生效

下一次运行 `enrich` 或生成领域级 `report` 时就会生效。

如果你想覆盖旧的逐篇总结，推荐这样跑：

```bash
python3 main.py --config config/config-poe.json enrich --limit 20 --with-llm --extra-config config/config-ikun.json --workers 1 --force
```

## 17. 常见问题

### 17.1 为什么有些论文显示“仅基于摘要/元数据”

常见原因：

- PDF 下载失败
- 全文抽取失败
- 当前后端不支持 PDF 输入
- 你运行时使用了 `--skip-pdf`

### 17.2 为什么两个模型对同一篇论文的输入依据不同

因为每个后端能力不同：

- 某个模型可能支持直接 PDF
- 另一个模型可能只支持文本输入

所以这两条总结可以一个是：

- `已直接读取 PDF`

另一个是：

- `已读取完整全文`

这是正常现象。

### 17.3 如果我想重新从空库开始

删除数据库后重新初始化即可。

如果你已经确认要清空：

```bash
rm -f data/papers.db

rm -f artifacts/pdfs/*
rm -f artifacts/text/*

rm -f reports/daily/*
rm -f reports/weekly/*
rm -f reports/catalog/*
rm -f reports/papers/*
rm -f exports/*.json
rm -f exports/papers/*

python3 main.py init --force
```

然后再按“第一次使用”那一节重新抓取和增强。
