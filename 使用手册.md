# 论文监控程序使用手册

## 1. 这套程序能做什么

这是一套本地运行的论文监控工具，当前默认监控两个方向：

- `matrix_free_fem`
  - 有限元分析 Matrix-Free 算法优化
- `ai_operator_acceleration`
  - AI 算子加速、GEMM/Attention/编译器/硬件优化

它可以完成这些事情：

- 从 `arXiv`、`DBLP` 抓取论文元数据
- 去重、主题匹配、相关性打分
- 下载 PDF，抽取全文文本
- 让一个或多个 LLM 生成逐篇中文总结
- 生成日报、周报、论文库总览
- 为每篇论文导出独立的 Markdown / HTML / JSON 报告
- 手动新增、查找、删除论文

运行时会显示实时进度条，包括：

- 抓取进度
- PDF 下载与全文抽取进度
- 单篇 LLM 总结进度
- 日报 / 周报 / 总览生成进度

如果你只想快速查命令，不想先读完整手册，可以直接看：

- `命令速查表.md`

## 2. 先记住这 3 个推荐工作流

如果你不想先读完整手册，只记住下面 3 套流程就够了。

### 2.1 首次初始化：抓两个领域各 100 篇，并让 Ikun 深读

```bash
source .codex.env
python3 main.py --config config/config-poe.json fetch --source arxiv --source dblp --recent-limit 100 --page-size 50
python3 main.py --config config/config-ikun.json enrich --limit 200 --with-llm --workers 2
python3 main.py --config config/config-poe.json catalog-report --with-llm --extra-config config/config-ikun.json --digest-primary-only
python3 main.py --config config/config-poe.json report --type weekly --days 7 --with-llm --extra-config config/config-ikun.json --digest-primary-only
```

这套流程的含义是：

- 先抓两个领域最近的论文
- 首次全量逐篇总结优先用 `Ikun / gpt-5.4`
- 总览和周报里的领域级概览由 `Poe / Claude` 负责

### 2.2 日常增量：新增论文双模型对比，领域报告只让 Poe 写

```bash
source .codex.env
python3 main.py --config config/config-poe.json run-once --type daily --source arxiv --source dblp --page-size 20 --since-last-run --enrich --enrich-limit 20 --with-llm --extra-config config/config-ikun.json --workers 2 --digest-primary-only
```

这条命令会：

- 只抓自上次成功抓取后新增的论文
- 对新增论文同时生成 `Poe` 和 `Ikun` 两份单篇总结
- 生成当日日报
- 领域级聚合总结只由主模型 `Poe` 生成

### 2.3 新增论文很多时：省 Ikun 配额

```bash
source .codex.env
python3 main.py --config config/config-poe.json enrich --limit 20 --with-llm --extra-config config/config-ikun.json --workers 2 --since-last-run --secondary-on-priority --secondary-top-per-topic 5 --secondary-min-score 26
```

这时：

- `Poe` 仍然处理全部新增论文
- `Ikun` 只处理高优先级论文

## 3. 环境要求

推荐环境：

- `Python 3.11+`
- Linux / macOS / WSL

推荐系统工具：

- `pdftotext`
- `pdfinfo`

如果系统里有 `pdftotext / pdfinfo`，全文抽取会更稳定。

## 4. 配置 API Key

推荐在项目根目录创建一个本地环境文件：

```bash
cat > .codex.env <<'EOF'
export POE_API_KEY='你的 Poe API Key'
export IKUNCODE_API_KEY='你的 Ikun API Key'
export SAURLAX_API_KEY='你的 SAURLAX API Key'
EOF
```

每次运行前先加载：

```bash
source .codex.env
```

注意：

- 不要把真实密钥直接写进配置文件里的 `api_key_env`
- `api_key_env` 里应该写环境变量名，不是密钥本身

## 5. 配置文件怎么理解

当前最常用的配置文件有 4 份：

- `config/config-poe.json`
  - 主配置
  - 当前模型：`claude-opus-4.6`
  - 当前更适合：领域聚合、日报、周报、总览、后续新增论文的第一视角
- `config/config-ikun.json`
  - 第二模型
  - 当前模型：`gpt-5.4`
  - 当前更适合：单篇深读、首次全量逐篇总结、重点论文精读
- `config/config-SAURLAX.json`
  - 可选旧配置
- `config/config.example.json`
  - 纯模板
  - 不直接连接任何真实后端

### 5.1 哪些字段最重要

最常改的是：

- `topics`
  - 抓取关键词和打分规则
- `llm.base_url`
- `llm.api_key_env`
- `llm.model`
- `llm.pdf_input_mode`
- `llm.max_output_tokens`
- `llm.model_reasoning_effort`
- `llm.model_output_effort`
- `llm.reasoning_by_task`
- `llm.output_effort_by_task`

### 5.2 现在两个主模型怎么配置

当前建议：

- `config/config-poe.json`
  - `pdf_input_mode = force`
  - `model_output_effort = max`
  - `output_effort_by_task.topic_digest = max`
- `config/config-ikun.json`
  - `pdf_input_mode = force`
  - `model_reasoning_effort = xhigh`
  - `reasoning_by_task.paper_summary = xhigh`

## 7. `init`：初始化目录和模板

```bash
python3 main.py init
python3 main.py init --force
```

作用：

- 初始化目录
- 生成模板配置
- 初始化数据库表结构

注意：

- `init --force` 不是“清空工作区”
- 它不会删除数据库内容、PDF 缓存、报告文件

## 8. `fetch`：只抓论文，不下载 PDF

最常用：

```bash
python3 main.py --config config/config-poe.json fetch --source arxiv --source dblp --recent-limit 100 --page-size 50
```

这条命令会：

- 从 `arXiv` 和 `DBLP` 抓论文元数据
- 做去重、主题匹配、相关性打分
- 写入数据库

它不会做：

- 下载 PDF
- 抽取全文
- 生成 LLM 总结

### 8.1 常用参数

- `--source`
  - 选择抓取源
- `--recent-limit`
  - 每个领域保留最近多少篇候选论文
- `--page-size`
  - 单次请求分页大小
- `--start-year`
  - 首次回填时只保留不早于该年份的论文
- `--since-last-run`
  - 只抓自上次成功抓取后新增的论文

## 9. `enrich`：下载 PDF、抽全文、生成逐篇总结

这是最重要的命令。

### 9.1 首次全量用 Ikun 深读

```bash
python3 main.py --config config/config-ikun.json enrich --limit 200 --with-llm --workers 2
```

### 9.2 新增论文双模型一起做

```bash
python3 main.py --config config/config-poe.json enrich --limit 20 --with-llm --extra-config config/config-ikun.json --workers 2 --since-last-run
```

### 9.3 新增论文很多时，只让 Ikun 处理高优先级论文

```bash
python3 main.py --config config/config-poe.json enrich --limit 20 --with-llm --extra-config config/config-ikun.json --workers 2 --since-last-run --secondary-on-priority --secondary-top-per-topic 5 --secondary-min-score 26
```

### 9.4 这条命令会做什么

- 下载 PDF
- 抽取全文文本
- 如果后端支持，直接把 PDF 输入给模型
- 如果后端不支持或直接 PDF 请求失败，自动回退到全文文本模式
- 生成逐篇 LLM 总结

关于速度：

- 现在 `--workers` 对完整 `enrich` 主流程已经生效
- 也就是说，`下载 PDF + 抽全文 + LLM 总结` 都可以并发执行
- 提高 `workers` 不会降低单篇总结质量，它只会并行处理多篇论文
- 但并发越高，越容易遇到后端限流、超时，token 消耗也会更快
- 对 `Ikun`，建议先从 `--workers 2` 开始；稳定后再尝试 `3-4`

### 9.5 常用参数

- `--limit`
  - 本轮最多处理多少篇论文
- `--topic`
  - 只处理某个领域
- `--paper-id`
  - 只处理指定论文
- `--with-llm`
  - 启用 LLM 总结
- `--extra-config`
  - 再加一份额外模型配置
- `--workers`
  - 并发数
  - 现在作用于完整 `enrich` 流程，不再只是部分场景
  - 推荐先用 `2`
- `--force`
  - 忽略缓存，强制重跑
- `--skip-pdf`
  - 跳过 PDF 和全文，只基于标题/摘要总结
- `--since-last-run`
  - 只增强自上次成功增强后新增入库的论文
- `--secondary-on-priority`
  - 额外模型只处理高优先级论文
- `--secondary-top-per-topic`
  - 每个领域额外送给第二模型的 Top N
- `--secondary-min-score`
  - 分数高于这个阈值的论文也送给第二模型

### 9.6 什么时候应该用哪个模型

推荐策略：

- 首次全量逐篇总结：优先 `Ikun`
- 后续新增论文：`Poe + Ikun` 一起做
- 新增论文很多：`Poe` 全量，`Ikun` 只看高优先级
- 只想让 `Ikun` 深读一篇：直接用 `config/config-ikun.json`

## 10. `report`：生成日报或周报

### 10.1 日报

```bash
python3 main.py --config config/config-poe.json report --type daily --with-llm --extra-config config/config-ikun.json --digest-primary-only
```

### 10.2 周报

```bash
python3 main.py --config config/config-poe.json report --type weekly --days 7 --with-llm --extra-config config/config-ikun.json --digest-primary-only
```

### 10.3 `--digest-primary-only` 是什么

这很重要。

加上它以后：

- 报告里仍然可以显示多模型的单篇摘要
- 但领域级聚合总结只由主配置模型生成

在你现在的工作流里，这通常意味着：

- 单篇摘要可以有 `Poe + Ikun`
- 领域日报 / 周报只让 `Poe` 来写

## 18. `chat_with_config.py`：先和模型直接聊天

这个脚本不会抓论文，也不会写数据库。

它只是读取某个配置文件，直接和该模型对话。

### 18.1 交互式聊天

```bash
source .codex.env
python3 chat_with_config.py --config config/config-poe.json
python3 chat_with_config.py --config config/config-ikun.json
```

### 18.2 单次提问

```bash
python3 chat_with_config.py --config config/config-poe.json --message "请解释 Tensor Core 和普通 CUDA Core 的区别"
```

### 18.3 多行输入

```bash
python3 chat_with_config.py --config config/config-poe.json --message $'请帮我做三件事：\n1. 总结这篇论文\n2. 提取关键方法\n3. 给出局限'
```

### 18.4 不使用系统提示词

```bash
python3 chat_with_config.py --config config/config-poe.json --system "" --message "你的问题"
```

### 18.5 看实际 payload

```bash
python3 chat_with_config.py --config config/config-poe.json --message "你好" --verbose
```

这个模式适合：

- 测试新接入的 API URL
- 测试 `reasoning_effort / output_effort / thinking_level`
- 先调 prompt，再决定是否放入主流程

## 20. PDF 总结逻辑是怎样的

当前优先级是：

1. 如果本地有 PDF，优先尝试把 PDF 直接输入给模型
2. 如果后端不支持或失败，再回退到全文文本模式
3. 如果没有全文，再回退到摘要 + 元数据模式

当前两条常用后端的具体逻辑是：

- `Poe / Claude`
  - 优先直接走 `PDF -> chat_file`
- `Ikun / gpt-5.4`
  - 优先尝试 `PDF -> 结构化直读`
  - 如果这一步失败，再尝试 `PDF -> pdf_brief`
  - 如果直接 PDF 仍失败，最后才回退到全文文本

### 20.1 怎么判断这次是不是直接读了 PDF

最简单的方法：

```bash
python3 main.py --config config/config-poe.json paper-report --paper-id 95 --extra-config config/config-ikun.json
```

然后看单篇报告里的：

- `输入依据`
- `依据说明`

如果看到：

- `已直接读取 PDF`

说明这次是直接读 PDF。

如果你看到：

- `direct_pdf_status = request_failed`
- `全文回退/request_failed`

表示本地 PDF 是有的，但“直接把 PDF 发给模型”这一步失败了，所以程序回退到了全文文本模式。

当前最常见的回退原因是：

- 后端长时间无响应或超时
- 后端返回空内容
- 直接 PDF 请求失败
- PDF 文件过大，超过 `pdf_inline_max_bytes`

## 21. prompts 提示词分别在什么场景使用

提示词文件在 `prompts/`。

### 21.1 `paper_summary_*`

用于单篇论文最终总结。

在你现在的推荐工作流里：

- 首次全量 `100 + 100` 篇更适合让 `Ikun` 跑这组提示词
- 后续新增论文如果启用双模型，`Poe` 和 `Ikun` 都会各跑一份

这一组现在更偏“深读型”：

- 强调优化对象、方法细节、硬件/软件栈、性能数字、局限
- `summary` 比以前更长
- 更适合单篇精读

### 21.2 `pdf_brief`

这不是单独文件，而是程序里的一个“轻量 PDF 全文备忘录”步骤。

它更适合：

- `Poe + Claude` 这类 `chat/completions + PDF` 后端

现在这一步也比以前更长、更结构化了：

- 固定输出 `摘要 / 问题 / 方法 / 应用 / 结果 / 贡献 / 局限 / 标签`
- 尽量保留平台、框架、基线、定量数字

### 21.3 `paper_chunk_*`

用于“全文文本分块分析”。

只有在：

- 模型不能直接读 PDF
- 但本地已经有全文文本

时才会用到。

### 21.4 `paper_reduce_*`

用于把全文分块分析结果聚合成最终单篇总结。

### 21.5 `topic_digest_*`

用于：

- 日报里的领域总结
- 周报里的领域总结
- 总览里的领域级概览

在你当前推荐的工作流里：

- 领域级聚合总结优先让 `Poe` 来写
- 如果命令里加了 `--digest-primary-only`，就只会由主配置模型生成领域总结

这一组现在也更适合真正的日报 / 周报：

- `overview` 更完整
- `highlights` 和 `watchlist` 更多
- 更强调“为什么值得继续跟”

## 22. 我应该改哪些关键词

- `topics[].priority_arxiv_categories`
- `topics[].dblp_venue_keywords`
- `topics[].priority_venue_keywords`
- `topics[].threshold`

## 23. 如何接入新的 LLM URL

通常不需要改代码，只要新增一份配置文件。

至少要改这些字段：

- `llm.variant_id`
- `llm.label`
- `llm.provider`
- `llm.base_url`
- `llm.api_key_env`
- `llm.model`

可选再改：

- `llm.pdf_input_mode`

- `llm.model_reasoning_effort`
- `llm.model_output_effort`

## 24. 常见问题

### 24.1 `fetch` 会下载 PDF 吗

不会。

- `fetch` 只抓元数据
- `enrich` 才会下载 PDF 和做全文总结

### 24.2 为什么数据库空了，但 `artifacts/pdfs` 里还有文件

因为清数据库不会自动删缓存文件。

数据库、PDF 缓存、全文文本、旧报告是四类不同的数据。

### 24.3 `paper-add` 手动加的论文也有 `paper_id` 吗

有。

命令执行后会直接打印。
