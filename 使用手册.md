# 论文监控程序使用手册

## 1. 这套程序能做什么

这是一套本地运行的论文监控工具，当前默认监控两个方向：

- `matrix_free_fem`
  - 有限元分析 Matrix-Free 算法优化
- `ai_operator_acceleration`
  - AI 算子加速、GEMM/Attention/编译器/硬件优化

它可以完成这些事情：

- 从 `arXiv`、`DBLP` 抓取论文元数据
- 去重、主题匹配、相关性打分
- 首次执行时把一批预置种子论文直接写入数据库
- 在抓取候选里混合保留“近期论文”和“高引用经典论文”，不再只按年份筛
- 下载 PDF，抽取全文文本
- 通过 landing page 和 `Semantic Scholar openAccessPdf` 补全一部分原始元数据里没有直接 PDF 链接的论文
- 让一个或多个 LLM 生成逐篇中文总结
- 生成日报、周报、论文库总览
- 当日报 / 周报窗口内没有严格命中的新增论文时，自动补充近 `30/90` 天高分论文回顾
- 为每篇论文导出独立的 Markdown / HTML / JSON 报告
- 手动新增、查找、删除论文

运行时会显示实时进度条，包括：

- 抓取进度
- PDF 下载与全文抽取进度
- 单篇 LLM 总结进度
- 日报 / 周报 / 总览生成进度

如果你只想快速查命令，不想先读完整手册，可以直接看：

- `命令速查表.md`

## 2. 先记住这 3 个推荐工作流

如果你不想先读完整手册，只记住下面 3 套流程就够了。

### 2.1 首次初始化：抓两个领域各 100 篇，并让 Ikun 深读

```bash
source .codex.env
python3 main.py --config config/config-poe.json fetch --source arxiv --source dblp --recent-limit 100 --page-size 50
python3 main.py --config config/config-ikun.json enrich --limit 200 --with-llm --workers 2
python3 main.py --config config/config-poe.json catalog-report --with-llm --extra-config config/config-ikun.json --digest-primary-only
python3 main.py --config config/config-poe.json report --type weekly --days 7 --with-llm --extra-config config/config-ikun.json --digest-primary-only
```

这套流程的含义是：

- 先抓两个领域最近的论文
- 首次全量逐篇总结优先用 `Ikun / gpt-5.4`
- 总览和周报里的领域级概览由 `Poe / Claude` 负责

### 2.2 日常增量：新增论文双模型对比，领域报告只让 Poe 写

```bash
source .codex.env
python3 main.py --config config/config-poe.json run-once --type daily --source arxiv --source dblp --page-size 20 --since-last-run --enrich --enrich-limit 20 --with-llm --extra-config config/config-ikun.json --workers 2 --digest-primary-only
```

这条命令会：

- 只抓自上次成功抓取后新增的论文
- 对新增论文同时生成 `Poe` 和 `Ikun` 两份单篇总结
- 生成当日日报
- 领域级聚合总结只由主模型 `Poe` 生成

### 2.3 新增论文很多时：省 Ikun 配额

```bash
source .codex.env
python3 main.py --config config/config-poe.json enrich --limit 20 --with-llm --extra-config config/config-ikun.json --workers 2 --since-last-run --secondary-on-priority --secondary-top-per-topic 5 --secondary-min-score 26
```

这时：

- `Poe` 仍然处理全部新增论文
- `Ikun` 只处理高优先级论文

### 2.4 用 Poe 专门补跑 Ikun 回退或缺失的论文

```bash
source .codex.env
python3 main.py --config config/config-poe.json enrich --limit 200 --with-llm --workers 2 --retry-from-variant ikun --retry-from-status incomplete
```

这时：

- 当前执行模型是 `Poe`
- 只挑那些 `Ikun` 结果还不完整的论文
- `incomplete` 表示：
  - `Ikun` 还没有摘要
  - 或 `Ikun` 只有 fallback 结果，不是完整的 `pdf_direct`

## 3. 环境要求

推荐环境：

- `Python 3.11+`
- Linux / macOS / WSL

推荐系统工具：

- `pdftotext`
- `pdfinfo`

如果系统里有 `pdftotext / pdfinfo`，全文抽取会更稳定。

## 4. 配置 API Key

推荐在项目根目录创建一个本地环境文件：

```bash
cat > .codex.env <<'EOF'
export POE_API_KEY='你的 Poe API Key'
export IKUNCODE_API_KEY='你的 Ikun API Key'
export SAURLAX_API_KEY='你的 SAURLAX API Key'
export SEMANTIC_SCHOLAR_API_KEY='可选：Semantic Scholar API Key'
EOF
```

每次运行前先加载：

```bash
source .codex.env
```

注意：

- 不要把真实密钥直接写进配置文件里的 `api_key_env`
- `api_key_env` 里应该写环境变量名，不是密钥本身
- `SEMANTIC_SCHOLAR_API_KEY` 不是必须
  - 不配置时，程序仍会尝试 `Semantic Scholar` 的 `DOI/arXiv details` 接口
  - 配了以后，标题搜索补 PDF 会更稳，也更不容易遇到 `429`

## 5. 配置文件怎么理解

当前最常用的配置文件有 4 份：

- `config/config-poe.json`
  - 主配置
  - 当前模型：`claude-opus-4.6`
  - 当前更适合：领域聚合、日报、周报、总览、后续新增论文的第一视角
- `config/config-ikun.json`
  - 第二模型
  - 当前模型：`gpt-5.4`
  - 当前更适合：单篇深读、首次全量逐篇总结、重点论文精读
- `config/config-SAURLAX.json`
  - 可选旧配置
- `config/config.example.json`
  - 纯模板
  - 不直接连接任何真实后端

### 5.1 哪些字段最重要

最常改的是：

- `topics`
  - 抓取关键词和打分规则
- `llm.base_url`
- `llm.api_key_env`
- `llm.model`
- `llm.pdf_input_mode`
- `llm.max_output_tokens`
- `llm.model_reasoning_effort`
- `llm.model_output_effort`
- `llm.reasoning_by_task`
- `llm.output_effort_by_task`

### 5.2 现在两个主模型怎么配置

当前建议：

- `config/config-poe.json`
  - `pdf_input_mode = force`
  - `model_output_effort = max`
  - `output_effort_by_task.topic_digest = max`
- `config/config-ikun.json`
  - `pdf_input_mode = force`
  - `model_reasoning_effort = xhigh`
  - `reasoning_by_task.paper_summary = xhigh`

### 5.3 `pdf_inline_max_bytes` 为什么不直接设很大

这个值可以在 config 里改，但不建议无脑调得特别大。

原因是：

- PDF 会先被转成 `base64 data URL`
- 真正发给后端的请求体会比原始 PDF 更大
- 值设得太大后，更容易触发：
  - 网关请求体大小限制
  - 反向代理限制
  - 上传超时
  - 单次请求特别慢

所以它不是“越大越好”，而是“够用就好”。

建议：

- 先保持当前值
- 如果日志明确出现 `pdf_too_large`，再逐步调大
- 不要一次放大很多倍

### 5.4 目前整个抓取和补 PDF 的流程

这套程序现在的完整流程是：

1. `fetch`
   - 从 `arXiv`、`DBLP` 抓论文元数据
   - 自动注入预置种子论文
   - 先按领域硬性条件过滤
   - 再综合关键词、引用、时间做打分
   - 在 `recent-limit` 窗口内混合保留“高引用经典论文”和“近期论文”
   - 把论文先写进本地数据库
2. `enrich`
   - 为每篇论文尝试找到可下载的 PDF
   - 下载 PDF
   - 抽取全文文本
   - 让 LLM 生成单篇总结
3. `report`
   - 基于数据库里的论文和单篇总结，生成日报、周报、总览和单篇报告

其中最容易让人误解的是第 2 步的“找 PDF”。现在程序按下面的顺序尝试：

1. 论文元数据里原本就有 `pdf_url`
2. 从 `DOI / arXiv / CoRR` 线索反推出 `arXiv PDF`
3. 如果是 arXiv `abs` 页面，直接改写成 `pdf` 链接
4. 抓论文 landing page，查这些线索：
   - `citation_pdf_url`
   - `dc.identifier`
   - `og:url`
   - 页面里的 `.pdf` 链接
5. 查询 `Semantic Scholar`：
   - 优先用 `DOI` 或 `arXiv ID` 的 details 接口读 `openAccessPdf`
   - 如果配置了 `SEMANTIC_SCHOLAR_API_KEY`，还会补做标题搜索
6. 如果还找不到，再尝试把 `primary_url` 当作 PDF 候选地址

只有真正找到可下载的 PDF 后，才会继续：

1. 下载 PDF
2. 抽取全文文本
3. 如果 PDF 大小不超过 `pdf_inline_max_bytes`
   - 直接把 PDF 喂给模型
4. 如果 PDF 太大，或直读失败
   - 回退到全文文本总结
5. 如果全文也失败
   - 最后才回退到 `abstract_metadata`

所以：

- 论文会先进入数据库，再去补 PDF
- 数据库里的论文，不一定都有本地 PDF
- `no-pdf` 不等于“不相关”，只表示当前没有拿到可下载全文

### 5.5 `Semantic Scholar` 现在能补回什么，不能补回什么

当前已经验证过：

- landing page 补回成功的论文：
  - `76`
  - `188`
- `Semantic Scholar -> openAccessPdf` 补回成功的论文：
  - `89`

同时也验证过，`Semantic Scholar` 不是万能的：

- `77`
  - `Semantic Scholar` 能给出 ACM 的 PDF 地址
  - 但源站返回 `403`
  - 所以仍然会保留 `no-pdf`
- `90`
  - landing page 能给出 Springer 的 `citation_pdf_url`
  - 但实际返回的是 HTML 页面，不是 PDF
- 还有一些 DOI 论文，`Semantic Scholar` 本身就没有返回 `openAccessPdf`

所以 `Semantic Scholar` 的定位是：

- 补充 PDF 发现源
- 不是主论文源
- 也不能保证所有期刊/出版社页面都能绕过权限或反爬

## 7. `init`：初始化目录和模板

```bash
python3 main.py init
python3 main.py init --force
```

作用：

- 初始化目录
- 生成模板配置
- 初始化数据库表结构

注意：

- `init --force` 不是“清空工作区”
- 它不会删除数据库内容、PDF 缓存、报告文件

## 8. `fetch`：只抓论文，不下载 PDF

最常用：

```bash
python3 main.py --config config/config-poe.json fetch --source arxiv --source dblp --recent-limit 100 --page-size 50
```

这条命令会：

- 从 `arXiv` 和 `DBLP` 抓论文元数据
- 自动把预置种子论文加入数据库
- 做去重、主题匹配、相关性打分
- 在 `--recent-limit 100` 这类场景下，不再是“纯最近 100 篇”
  - 会优先保留一部分高引用经典论文
  - 同时保留最近的新论文
- 写入数据库

它不会做：

- 下载 PDF
- 抽取全文
- 生成 LLM 总结

### 8.1 常用参数

- `--source`
  - 选择抓取源
- `--recent-limit`
  - 每个领域保留多少篇候选论文
  - 现在它是“混合窗口”，不是纯时间窗口
  - 例如 `100` 通常会尽量保留约一半高引用经典论文，再补足近期论文
- `--page-size`
  - 单次请求分页大小
- `--start-year`
  - 首次回填时只保留不早于该年份的论文
- `--since-last-run`
  - 只抓自上次成功抓取后新增的论文

## 9. `enrich`：下载 PDF、抽全文、生成逐篇总结

这是最重要的命令。

### 9.1 首次全量用 Ikun 深读

```bash
python3 main.py --config config/config-ikun.json enrich --limit 200 --with-llm --workers 2
```

### 9.2 新增论文双模型一起做

```bash
python3 main.py --config config/config-poe.json enrich --limit 20 --with-llm --extra-config config/config-ikun.json --workers 2 --since-last-run
```

### 9.3 新增论文很多时，只让 Ikun 处理高优先级论文

```bash
python3 main.py --config config/config-poe.json enrich --limit 20 --with-llm --extra-config config/config-ikun.json --workers 2 --since-last-run --secondary-on-priority --secondary-top-per-topic 5 --secondary-min-score 26
```

### 9.4 用 Poe 专门补跑 Ikun 回退或缺失的论文

```bash
python3 main.py --config config/config-poe.json enrich --limit 200 --with-llm --workers 2 --retry-from-variant ikun --retry-from-status incomplete
```

### 9.5 这条命令会做什么

- 下载 PDF
- 抽取全文文本
- 如果后端支持，直接把 PDF 输入给模型
- 如果后端不支持或直接 PDF 请求失败，自动回退到全文文本模式
- 生成逐篇 LLM 总结

关于速度：

- 现在 `--workers` 对完整 `enrich` 主流程已经生效
- 也就是说，`下载 PDF + 抽全文 + LLM 总结` 都可以并发执行
- 提高 `workers` 不会降低单篇总结质量，它只会并行处理多篇论文
- 但并发越高，越容易遇到后端限流、超时，token 消耗也会更快
- 对 `Ikun`，建议先从 `--workers 2` 开始；稳定后再尝试 `3-4`

### 9.6 常用参数

- `--limit`
  - 本轮最多处理多少篇论文
- `--topic`
  - 只处理某个领域
- `--paper-id`
  - 只处理指定论文
- `--with-llm`
  - 启用 LLM 总结
- `--extra-config`
  - 再加一份额外模型配置
- `--workers`
  - 并发数
  - 现在作用于完整 `enrich` 流程，不再只是部分场景
  - 推荐先用 `2`
- `--force`
  - 忽略缓存，强制重跑
- `--skip-pdf`
  - 跳过 PDF 和全文，只基于标题/摘要总结
- `--since-last-run`
  - 只增强自上次成功增强后新增入库的论文
- `--secondary-on-priority`
  - 额外模型只处理高优先级论文
- `--secondary-top-per-topic`
  - 每个领域额外送给第二模型的 Top N
- `--secondary-min-score`
  - 分数高于这个阈值的论文也送给第二模型
- `--retry-from-variant`
  - 只处理另一个模型仍缺失或回退的论文
  - 例如 `ikun`、`ikun_gpt-5.4`
- `--retry-from-status`
  - `incomplete`
    - 该模型缺失或回退都处理
  - `fallback`
    - 只处理该模型已有摘要但不是完整 `pdf_direct` 的论文
  - `missing`
    - 只处理该模型还没有摘要的论文

### 9.7 什么时候应该用哪个模型

推荐策略：

- 首次全量逐篇总结：优先 `Ikun`
- 后续新增论文：`Poe + Ikun` 一起做
- 新增论文很多：`Poe` 全量，`Ikun` 只看高优先级
- 只想让 `Ikun` 深读一篇：直接用 `config/config-ikun.json`
- `Ikun` 有一批回退或缺失结果时：让 `Poe` 用 `--retry-from-variant ikun` 单独补跑

### 9.8 如果 `enrich` 中途中断了，下次会怎样

现在的行为是：

- 如果某篇论文某个模型上次已经得到完整 `pdf_direct` 结果
  - 下次同模型 `enrich` 会自动跳过
- 如果上次只是：
  - `全文回退`
  - `摘要回退`
  - `direct_pdf_status = request_failed`
  - 或者根本还没有该模型的结果
  - 下次会自动重试

所以现在重新执行 `enrich` 时，会优先补：

- 没有该模型总结的论文
- 上次没能成功直读 PDF 的论文

如果你要让另一个模型专门来补这些论文，可以直接这样做：

```bash
python3 main.py --config config/config-poe.json enrich --limit 200 --with-llm --workers 2 --retry-from-variant ikun --retry-from-status incomplete
```

常见组合：

- `--retry-from-variant ikun --retry-from-status incomplete`
  - 让 `Poe` 补跑所有 `Ikun` 缺失或回退的论文
- `--retry-from-variant ikun --retry-from-status fallback`
  - 只补跑 `Ikun` 已有摘要但不是完整 PDF 直读的论文
- `--retry-from-variant ikun --retry-from-status missing`
  - 只补跑 `Ikun` 完全没生成摘要的论文

## 10. `report`：生成日报或周报

### 10.1 日报

```bash
python3 main.py --config config/config-poe.json report --type daily --with-llm --extra-config config/config-ikun.json --digest-primary-only
```

如果当天严格命中为 `0`：

- `matrix_free_fem` 会自动补充近 `90` 天高分回顾
- `ai_operator_acceleration` 会自动补充近 `30` 天高分回顾

这不会放宽抓取规则，只是让日报在空窗期仍然能看到最近最值得回顾的高分论文。

### 10.2 周报

```bash
python3 main.py --config config/config-poe.json report --type weekly --days 7 --with-llm --extra-config config/config-ikun.json --digest-primary-only
```

如果最近 `7` 天严格命中为 `0`，周报也会自动补充高分回顾：

- `matrix_free_fem`：近 `90` 天
- `ai_operator_acceleration`：近 `30` 天

报告会明确区分：

- 严格命中新论文
- 自动补充的高分回顾

### 10.3 `--digest-primary-only` 是什么

这很重要。

加上它以后：

- 报告里仍然可以显示多模型的单篇摘要
- 但领域级聚合总结只由主配置模型生成

在你现在的工作流里，这通常意味着：

- 单篇摘要可以有 `Poe + Ikun`
- 领域日报 / 周报只让 `Poe` 来写

另外现在的显示规则是：

- 同一篇论文如果已经保存了多个模型的结果
- 单篇报告、总览、日报、周报里会一起显示
- 也就是说：
  - 先用 `Ikun` 跑过
  - 后来又用 `Poe` 跑过
  - 报告里会把两份结果都展示出来

另外：

- 日报 / 周报里的论文条目现在按“相关性分数从高到低”排序
- 如果窗口内没有严格命中新论文，会显示自动补充的高分回顾

## 11. `catalog-report`：生成论文库总览

最常用：

```bash
python3 main.py --config config/config-poe.json catalog-report --with-llm --extra-config config/config-ikun.json --digest-primary-only
```

它会生成：

- 论文库总览 Markdown
- 论文库总览 HTML
- 论文库总览 JSON
- 并顺带导出单篇论文报告

总览里可以看到：

- 每个领域有哪些论文
- 每篇论文的基本信息
- 每篇论文的多模型摘要
- 领域级概览

## 12. `paper-report`：导出单篇论文报告

```bash
python3 main.py --config config/config-poe.json paper-report --paper-id 95 --extra-config config/config-ikun.json
```

如果你不传 `--paper-id`，也可以按时间窗口导出一批：

```bash
python3 main.py --config config/config-poe.json paper-report --type daily --date 2026-03-16
```

## 13. `paper-add` / `paper-find` / `paper-delete`

### 13.1 手动加入一篇论文

```bash
python3 main.py --config config/config-poe.json paper-add \
  --topic ai_operator_acceleration \
```

## 20. PDF 总结逻辑是怎样的

当前优先级是：

1. 如果本地有 PDF，优先尝试把 PDF 直接输入给模型
2. 如果后端不支持或失败，再回退到全文文本模式
3. 如果没有全文，再回退到摘要 + 元数据模式

当前两条常用后端的具体逻辑是：

- `Poe / Claude`
  - 优先直接走 `PDF -> chat_file`
- `Ikun / gpt-5.4`
  - 优先尝试 `PDF -> 结构化直读`
  - 如果这一步失败，再尝试 `PDF -> pdf_brief`
  - 如果直接 PDF 仍失败，最后才回退到全文文本

另外现在程序还会自动处理一种常见兼容问题：

- 如果后端报
  - `Unknown parameter ... file`
- 程序会自动改用另一种 PDF 内容块格式再试一次
- 所以这类问题不需要你手动改配置

### 20.1 怎么判断这次是不是直接读了 PDF

最简单的方法：

```bash
python3 main.py --config config/config-poe.json paper-report --paper-id 95 --extra-config config/config-ikun.json
```

然后看单篇报告里的：

- `输入依据`
- `依据说明`

如果看到：

- `已直接读取 PDF`

说明这次是直接读 PDF。

如果你看到：

- `direct_pdf_status = request_failed`
- `全文回退/request_failed`

表示本地 PDF 是有的，但“直接把 PDF 发给模型”这一步失败了，所以程序回退到了全文文本模式。

当前最常见的回退原因是：

- 后端长时间无响应或超时
- 后端返回空内容
- 直接 PDF 请求失败
- PDF 文件过大，超过 `pdf_inline_max_bytes`

如果日志里看到：

- `pdf_too_large:实际大小>pdf_inline_max_bytes`

就说明不是模型本身失败，而是这篇 PDF 超过了当前允许内联上传的大小。

## 21. prompts 提示词分别在什么场景使用

提示词文件在 `prompts/`。

### 21.1 `paper_summary_*`

用于单篇论文最终总结。

在你现在的推荐工作流里：

- 首次全量 `100 + 100` 篇更适合让 `Ikun` 跑这组提示词
- 后续新增论文如果启用双模型，`Poe` 和 `Ikun` 都会各跑一份

这一组现在更偏“深读型”：

- 强调优化对象、方法细节、硬件/软件栈、性能数字、局限
- `summary` 比以前更长
- 更适合单篇精读

### 21.2 `pdf_brief`

这不是单独文件，而是程序里的一个“轻量 PDF 全文备忘录”步骤。

它更适合：

- `Poe + Claude` 这类 `chat/completions + PDF` 后端

现在这一步也比以前更长、更结构化了：

- 固定输出 `摘要 / 问题 / 方法 / 应用 / 结果 / 贡献 / 局限 / 标签`
- 尽量保留平台、框架、基线、定量数字

### 21.3 `paper_chunk_*`

用于“全文文本分块分析”。

只有在：

- 模型不能直接读 PDF
- 但本地已经有全文文本

时才会用到。

### 21.4 `paper_reduce_*`

用于把全文分块分析结果聚合成最终单篇总结。

### 21.5 `topic_digest_*`

用于：

- 日报里的领域总结
- 周报里的领域总结
- 总览里的领域级概览

在你当前推荐的工作流里：

- 领域级聚合总结优先让 `Poe` 来写
- 如果命令里加了 `--digest-primary-only`，就只会由主配置模型生成领域总结

这一组现在也更适合真正的日报 / 周报：

- `overview` 更完整
- `highlights` 和 `watchlist` 更多
- 更强调“为什么值得继续跟”

## 22. 我应该改哪些关键词

修改你实际运行时传给 `--config` 的那份配置文件。

如果你当前主用的是：

- `config/config-poe.json`

那么重点改这里：

- `topics[].source_queries`
- `topics[].required_keyword_groups`
- `topics[].must_match_groups`
- `topics[].positive_keywords`
- `topics[].exclude_keywords`
- `topics[].arxiv_categories`
- `topics[].priority_arxiv_categories`
- `topics[].dblp_venue_keywords`
- `topics[].priority_venue_keywords`
- `topics[].threshold`
- `topics[].seed_papers`

这几个字段的含义现在要这样理解：

- `required_keyword_groups`
  - 硬性条件
  - 少一组就直接判为不相关
- `must_match_groups`
  - 软性条件
  - 命中后加分，不命中会扣分，但不会单独决定淘汰
- `positive_keywords`
  - 额外加分
- `seed_papers`
  - 首次执行时直接注入数据库的奠基性论文

### 22.1 两个默认领域现在的筛选逻辑

- `matrix_free_fem`
  - 必须直接命中 `matrix-free / matrix free / matrix-free operator evaluation` 这类表述
  - 同时必须涉及实现或性能讨论
  - 仅仅是“有限元 + GPU”不会再进来
- `ai_operator_acceleration`
  - 必须同时命中“实现/性能优化”线索和“算子/编译器/硬件”线索
  - 纯模型设计、纯算法创新论文会被明显压低或排除

## 23. 如何接入新的 LLM URL

通常不需要改代码，只要新增一份配置文件。

至少要改这些字段：

- `llm.variant_id`
- `llm.label`
- `llm.provider`
- `llm.base_url`
- `llm.api_key_env`
- `llm.model`

可选再改：

- `llm.pdf_input_mode`
- `llm.max_output_tokens`
- `llm.model_reasoning_effort`
- `llm.model_output_effort`
- `llm.reasoning_by_task`
- `llm.output_effort_by_task`

## 24. 常见问题

### 24.1 `fetch` 会下载 PDF 吗

不会。

- `fetch` 只抓元数据
- `enrich` 才会下载 PDF 和做全文总结

### 24.2 为什么数据库空了，但 `artifacts/pdfs` 里还有文件

因为清数据库不会自动删缓存文件。

数据库、PDF 缓存、全文文本、旧报告是四类不同的数据。

### 24.3 `paper-add` 手动加的论文也有 `paper_id` 吗

有。

命令执行后会直接打印。

如果忘了，可以用：

```bash
python3 main.py --config config/config-poe.json paper-find --title "你的标题关键字"
```

### 24.4 `Poe` 和 `Ikun` 谁更适合什么

以你当前配置和实测结果为准：

- `Poe / Claude`
  - 更适合领域聚合、日报、周报、总览
  - 也适合作为后续新增论文的主模型
- `Ikun / gpt-5.4`
  - 更适合首次全量逐篇深读
  - 更适合重点论文精读和第二意见

### 24.5 我现在该怎么开始

如果你已经配好了 API Key，直接从这里开始：

```bash
source .codex.env
python3 main.py --config config/config-poe.json fetch --source arxiv --source dblp --recent-limit 100 --page-size 50
python3 main.py --config config/config-ikun.json enrich --limit 200 --with-llm --workers 2
python3 main.py --config config/config-poe.json catalog-report --with-llm --extra-config config/config-ikun.json --digest-primary-only
```
