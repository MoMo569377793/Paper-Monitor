# 论文监控程序使用手册

## 1. 这个程序能做什么

这是一套本地运行的论文监控工具，默认已经配置好两个方向：

- `有限元分析 Matrix-Free 算法优化`
- `AI 算子加速`

程序会自动完成下面几件事：

- 从 `arXiv`、`DBLP` 抓取新论文
- 做去重、主题匹配和相关性打分
- 下载 PDF，抽取全文文本
- 调用两个 LLM 分别生成每篇论文的中文结构化总结
- 生成日报、周报
- 额外为每篇论文单独生成 Markdown / HTML / JSON 报告

## 2. 现在这版和之前相比有什么变化

当前版本新增了三个关键能力：

1. `LLM 自动检测是否支持直接读 PDF`

- 如果后端支持直接 PDF 输入，程序会优先把整篇 PDF 直接交给模型
- 如果后端不支持，程序会自动回退为：
  - `完整 PDF -> 本地抽全文 -> 分块总结 -> 聚合`
- 如果连全文都没有，才会退到：
  - `标题 + 摘要 + 元数据`

2. `单篇论文总结单独可见`

- 不再只能在 `daily / weekly` 里看摘要
- 现在每篇论文都会单独导出到：
  - `reports/papers/`
  - `exports/papers/`

3. `报告中会明确写清 LLM 的输入依据`

每个模型摘要都会标注下面三种之一：

- `已直接读取 PDF`
- `已读取完整全文`
- `仅基于摘要/元数据`

## 3. 环境要求

- `Python 3.11+`
- Linux / macOS / WSL
- 推荐安装：
  - `pdftotext`
  - `pdfinfo`

程序主体只依赖 Python 标准库，但如果系统里有 `pdftotext / pdfinfo`，PDF 抽取质量会更稳定。

## 5. 先配置 API Key

推荐在项目根目录创建一个本地环境文件：

```bash
cat > .codex.env <<'EOF'
export SAURLAX_API_KEY='你的 deepseek-chat API Key'
export IKUNCODE_API_KEY='你的 gpt-5.4 API Key'
EOF
```

每次运行前先加载：

```bash
source /home/momo/git_ws/search/.codex.env
```

注意：

- `config/config.json` 默认使用 `deepseek-chat`
- `config/config.example.json` 默认使用 `gpt-5.4`
- 不要把真实密钥直接写进 `api_key_env`
- `api_key_env` 里应该写“环境变量名”，不是密钥本身

### 5.1 如果你要接入一个新的 LLM URL

通常不需要改代码，只需要新增一份配置文件。

最稳的做法是：

1. 复制一份现有配置
2. 只修改 `llm` 段里的几个字段
3. 再给这个新配置配一个环境变量

例如，你可以从现有配置复制一份：

```bash
cp config/config.example.json config/config.new.json
```

然后只改下面这些字段：

- `variant_id`
  - 必须唯一，建议写成 `config-new-backend`
- `label`
  - 报告里显示的名字
- `provider`
  - 如果新接口兼容 `POST /chat/completions`，填 `openai_compatible`
  - 如果新接口走 `POST /responses`，填 `openai_responses`
- `base_url`
  - 例如 `https://your-api.example.com/v1`
- `api_key_env`
  - 例如 `NEW_BACKEND_API_KEY`
- `model`
  - 例如 `deepseek-chat`、`gpt-5.4`

其他字段一般不用改，保留默认值即可。

一个最小示例如下：

```json
"llm": {
  "variant_id": "config-new-backend",
  "label": "new-backend",
  "enabled": true,
  "provider": "openai_compatible",
  "base_url": "https://your-api.example.com/v1",
  "api_key_env": "NEW_BACKEND_API_KEY",
  "model": "your-model-name",
  "timeout_seconds": 60,
  "temperature": 0.2,
  "max_input_chars": 16000,
  "max_output_tokens": 700,
  "store": false,
  "enable_topic_digest": true,
  "topic_digest_entry_limit": 8,
  "fulltext_chunk_chars": 10000,
  "fulltext_chunk_overlap_chars": 1200,
  "fulltext_max_chunks": 12,
  "pdf_input_mode": "auto",
  "pdf_inline_max_bytes": 12582912
}
```

然后在 `.codex.env` 里加上这个新 URL 对应的 key：

```bash
export NEW_BACKEND_API_KEY='你的新 API Key'
```

### 5.2 接入新 URL 以后怎么使用

有两种常见用法。

第一种：把它当成“主模型”单独跑。

```bash
source /home/momo/git_ws/search/.codex.env
python3 main.py --config config/config.new.json run-once --type daily --source arxiv --source dblp --enrich --with-llm
```

第二种：把它作为“额外模型”接到现有流程里一起对比。

```bash
source /home/momo/git_ws/search/.codex.env
python3 main.py --config config/config.json enrich --limit 20 --with-llm --extra-config config/config.new.json --workers 2
python3 main.py --config config/config.json report --type daily --with-llm --extra-config config/config.new.json
```

第二种更适合你现在这个项目，因为可以直接和现有模型放在同一份报告里比较。

### 5.3 怎么检测这个新 URL 是否支持 PDF 输入

程序本身已经会自动检测，不需要你手动写测试代码。

推荐按下面步骤验证：

1. 先确保某篇论文已经下载到本地 PDF
2. 用新配置跑一次带 LLM 的增强
3. 生成该论文的单篇报告
4. 看报告里的 `输入依据`

最直接的一组命令是：

```bash
source /home/momo/git_ws/search/.codex.env
python3 main.py --config config/config.new.json enrich --limit 1 --with-llm --workers 1
python3 main.py --config config/config.new.json paper-report --paper-id 133
```

然后打开生成的单篇报告，看每个模型摘要旁边的 `输入依据`：

- 如果显示 `已直接读取 PDF`
  - 说明这个 URL 支持 PDF 输入
- 如果显示 `已读取完整全文`
  - 说明这个 URL 不支持直接 PDF，程序已经自动回退到“抽全文文本后总结”
- 如果显示 `仅基于摘要/元数据`
  - 说明这次既没有成功直读 PDF，也没有拿到可用全文文本

### 5.4 接入新 URL 时最容易出错的地方

- `provider` 写错
  - 兼容接口通常用 `openai_compatible`
- `base_url` 少了 `/v1`
  - 很多网关要求带 `/v1`
- `api_key_env` 里直接写了真实 key
  - 这里应该写环境变量名，不是密钥本身
- `variant_id` 和已有配置重复
  - 重复后会导致多模型模式报错
- 新 URL 实际不支持 PDF
  - 这不是问题，程序会自动回退到全文文本模式

## 6. 最快上手

### 6.1 第一次使用：先抓论文，再做双模型总结

第 1 步，抓取两个领域的论文：

```bash
python3 main.py --config config/config.json fetch --source arxiv --source dblp --recent-limit 200 --page-size 50
```

说明：

- `--recent-limit 200` 表示先往深处抓一批候选
- 抓取时会显示实时进度条

第 2 步，用两个模型生成逐篇总结：

```bash
python3 main.py --config config/config.json enrich --limit 200 --with-llm --extra-config config/config.example.json --workers 2
```

说明：

- 这一步会先尝试下载 PDF
- 然后自动判断模型是否支持直接读 PDF
- 如果支持，就直接用 PDF 总结
- 如果不支持，就退回“全文文本分块总结”
- 如果你使用 `--skip-pdf`，程序将无法做真正的全文总结

第 3 步，生成日报：

```bash
python3 main.py --config config/config.json report --type daily --with-llm --extra-config config/config.example.json
```

第 4 步，生成周报：

```bash
python3 main.py --config config/config.json report --type weekly --days 7 --with-llm --extra-config config/config.example.json
```

## 10. 关于“直接读 PDF”和“抽全文后再总结”

程序现在会自动按下面顺序尝试：

1. `直接 PDF`
2. `完整全文文本`
3. `摘要 + 元数据`

### 10.1 什么情况下会直接用 PDF

同时满足下面条件时会尝试：

- 本地已经有 PDF 文件
- 当前 LLM 后端经探测支持 PDF 输入
- PDF 文件大小没有超过配置上限

### 10.2 什么情况下会回退到全文文本

出现下面任一情况就会回退：

- 后端不支持 PDF 输入
- 后端对 PDF 请求报错
- 本地 PDF 太大，不适合直接内联发送
- 当前没有可用 PDF，但已经抽取过全文文本

### 10.3 报告里如何看出用了哪种方式

在日报、周报、单篇报告里，每个模型摘要旁边都会写：

- `输入依据`
- `依据说明`

例如：

- `已直接读取 PDF`
- `本次总结由模型直接读取 PDF 文件 xxx.pdf 后生成`

或者：

- `已读取完整全文`
- `本次总结读取了完整 PDF 提取全文，并按 11 个分块进行分析后聚合`

## 11. 重要参数解释

### 11.1 抓取参数

- `--recent-limit`
  - 按时间顺序保留最近多少篇候选论文
- `--start-year`
  - 首次回填时，只保留不早于某一年的论文
- `--page-size`
  - 每次请求抓多少条
- `--since-last-run`
  - 只处理自上次成功运行后新增的论文

### 11.2 增强参数

- `--with-llm`
  - 启用模型总结
- `--extra-config config/config.example.json`
  - 接入第二个模型一起生成结果
- `--workers 2`
  - 并发处理的 worker 数
- `--skip-pdf`
  - 跳过 PDF 下载和全文抽取
  - 注意：这会让总结退化为摘要模式

### 11.3 报告参数

- `report --type daily`
  - 生成日报
- `report --type weekly --days 7`
  - 生成 7 天周报
- `paper-report --paper-id 133`
  - 只导出单篇论文报告

## 12. 配置文件里最重要的字段

### 12.1 `llm` 段

当前新增的两个字段是：

- `pdf_input_mode`
  - `auto`：自动探测后端是否支持 PDF 输入，推荐
  - `disable`：禁用直接 PDF，始终走全文文本/摘要回退
  - `force`：强制按“支持 PDF”处理，只建议在你非常确定后端支持时使用

- `pdf_inline_max_bytes`
  - 直接内联发送 PDF 的最大字节数
  - 超过这个值会自动回退到全文文本模式

示例：

```json
"llm": {
  "enabled": true,
  "provider": "openai_compatible",
  "base_url": "https://ai.saurlax.com/v1",
  "api_key_env": "SAURLAX_API_KEY",
  "model": "deepseek-chat",
  "pdf_input_mode": "auto",
  "pdf_inline_max_bytes": 12582912
}
```

### 12.2 `prompts` 段

你可以直接改提示词文件：

- `prompts/paper_summary_system.txt`
- `prompts/paper_summary_user.txt`
- `prompts/paper_chunk_system.txt`
- `prompts/paper_chunk_user.txt`
- `prompts/paper_reduce_system.txt`
- `prompts/paper_reduce_user.txt`
- `prompts/topic_digest_system.txt`
- `prompts/topic_digest_user.txt`

如果你想调整“每篇论文怎么总结”，主要改这两个：

- `prompts/paper_summary_system.txt`
- `prompts/paper_summary_user.txt`

## 14. 常见问题

### 14.1 为什么有些论文显示“仅基于摘要/元数据”

原因通常是：

- 本地没有成功下载 PDF
- 没有抽取出可用全文
- 当前后端不支持直接 PDF
- 你运行时用了 `--skip-pdf`

### 14.2 为什么同一篇论文两个模型的输入依据不一定一样

因为每个后端的能力不同：

- 某个模型可能支持直接 PDF
- 另一个模型可能不支持，于是退到全文文本

所以报告会分别标注，不会混在一起。

### 14.3 我修改了提示词，什么时候生效

下一次运行 `enrich` 或 `run-once --enrich` 时就会生效。

如果你希望重新覆盖旧摘要，使用：

```bash
python3 main.py --config config/config.json enrich --limit 20 --with-llm --extra-config config/config.example.json --workers 2 --force
```

### 14.4 如何验证当前后端到底支不支持 PDF

最简单的方法就是：

1. 保证某篇论文已经下载了本地 PDF
2. 运行 `enrich --with-llm`
3. 打开单篇报告
4. 看 `输入依据`

如果显示：

- `已直接读取 PDF`

说明这次后端直接吃了 PDF。

如果显示：

- `已读取完整全文`

说明这次走的是“本地抽全文后总结”的回退路径。
