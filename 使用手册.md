# 论文监控程序使用手册

## 1. 项目简介

本程序用于本地监控你关心的论文主题，当前默认已经配置好两个方向：

- `有限元分析 Matrix-Free 算法优化`
- `AI 算子加速`

支持的数据源和处理方式如下：

- `arXiv`：通过官方 API 直接抓取题目、摘要、分类和链接
- `DBLP`：通过官方搜索 API 补会议/期刊元数据
- `Google Scholar Alerts`：不直接爬取 Scholar 页面，而是解析官方 Alert 邮件

程序会把论文统一写入本地 `SQLite` 数据库，再生成：

- `Markdown` 报告
- `HTML` 报告
- `JSON` 导出

## 2. 功能概览

当前版本具备以下能力：

- 多数据源抓取
- 关键词规则打分
- 去重合并
- PDF 下载与全文文本抽取
- 自动生成机器摘要
- 多 LLM 逐篇中文结构化总结
- 多 LLM 主题聚合日报 / 周报
- 支持并排比较不同模型的摘要结果
- 本地保存日报 / 周报
- 支持持续轮询模式
- 支持 Google Scholar Alert 邮箱解析

## 3. 环境要求

- `Python 3.11+`
- Linux / macOS / WSL 均可
- 不依赖第三方 Python 库，直接用标准库即可运行
- 推荐安装 `pdftotext` 和 `pdfinfo`，这样 PDF 抽取效果更稳定

在当前环境中，程序已经验证过可直接调用系统里的 `pdftotext` / `pdfinfo`。

建议在项目根目录执行所有命令。

注意：全局参数如 `--config`、`--log-level` 需要写在子命令前面，例如：

```bash
python3 main.py --config /path/to/config.json report --type daily
```

## 5. 快速开始

### 5.1 先配置 API Key

推荐在项目根目录创建一个本地文件：

```bash
cat > .codex.env <<'EOF'
export SAURLAX_API_KEY='你的 deepseek-chat API Key'
export IKUNCODE_API_KEY='你的 gpt-5.4 API Key'
EOF
```

之后每次运行前先加载：

```bash
source /home/momo/git_ws/search/.codex.env
```

### 5.2 首次初始化：每个领域最新 100 篇 + 双模型总结

首次使用时，建议按下面 4 步跑。

第 1 步：抓取候选论文。

```bash
python3 main.py --config config/config.json fetch --source arxiv --source dblp --recent-limit 200 --page-size 50
```

说明：

- `--recent-limit 200` 不是最终报告只保留 200 篇，而是先抓深一点，再通过主题打分筛出每个领域前 100 篇有效论文
- 抓取阶段会显示实时进度条

第 2 步：给 `有限元分析 Matrix-Free 算法优化` 前 100 篇生成双模型全文详细介绍。

```bash
python3 main.py --config config/config.json enrich --limit 100 --topic matrix_free_fem --with-llm --extra-config config/config.example.json --workers 2
```

第 3 步：给 `AI 算子加速` 前 100 篇生成双模型全文详细介绍。

```bash
python3 main.py --config config/config.json enrich --limit 100 --topic ai_operator_acceleration --with-llm --extra-config config/config.example.json --workers 2
```

说明：

- 不加 `--skip-pdf` 时，程序会先下载 PDF、抽取完整正文，再让 LLM 基于整篇论文的分块分析结果生成最终总结
- `--workers 2` 更适合全文模式；如果开太大，网络和 LLM 请求更容易拥塞
- 增强阶段也会显示实时进度条

第 4 步：生成带双模型主题总结的周报。

```bash
python3 main.py --config config/config.json report --type weekly --days 7 --with-llm --extra-config config/config.example.json
```

输出位置：

- 周报 Markdown：`reports/weekly/YYYY-MM-DD.md`
- 周报 HTML：`reports/weekly/YYYY-MM-DD.html`
- 周报 JSON：`exports/weekly-YYYY-MM-DD.json`

### 5.3 短间隔增量抓取

如果你过一会儿想补抓“自上次成功运行后新增、且数据库里还没有的新论文”，建议这样跑：

```bash
python3 main.py --config config/config.json fetch --source arxiv --source dblp --page-size 20 --since-last-run
python3 main.py --config config/config.json enrich --limit 20 --with-llm --extra-config config/config.example.json --workers 2 --since-last-run
```

`--since-last-run` 会读取数据库 checkpoint。第一次使用这个参数时，由于还没有 checkpoint，会按当前抓取范围做一次初始化；后续再运行时，就只处理自上次成功抓取 / 增强后新增的论文。

### 5.4 每日自动执行一次

如果你想每天手动触发一次完整流程，用这一条：

```bash
python3 main.py --config config/config.json run-once --type daily --source arxiv --source dblp --page-size 20 --since-last-run --enrich --enrich-limit 20 --with-llm --extra-config config/config.example.json --workers 2
```

这条命令会连续显示 3 段进度条：

- 抓取进度条
- 增量增强进度条
- 日报生成进度条

输出位置：

- 日报 Markdown：`reports/daily/YYYY-MM-DD.md`
- 日报 HTML：`reports/daily/YYYY-MM-DD.html`
- 日报 JSON：`exports/daily-YYYY-MM-DD.json`

### 5.5 重要参数说明

- `--recent-limit`：抓取阶段每个领域向数据源回溯的候选深度
- `--page-size`：每次请求抓多少条
- `--since-last-run`：只处理自上次成功运行后新增的论文
- `--with-llm`：启用模型总结
- `--extra-config config/config.example.json`：把第二个模型 `gpt-5.4` 接进来
- `--skip-pdf`：跳过 PDF 下载。注意：这会让 LLM 无法阅读整篇论文，只能基于标题、摘要和元数据总结
- `--workers 2`：全文模式建议从 2 开始，避免并发过高导致下载或 API 拥塞

如果你希望使用 `paper-monitor` 命令：

## 6. 默认配置说明

默认配置文件位于：

```text
config/config.json
```

程序已经为你的两个主题预置了：

- `source_queries`
- `must_match_groups`
- `positive_keywords`
- `exclude_keywords`
- `arxiv_categories`
- `DBLP venue 关键词`

### 6.1 关键字段解释

- `database_path`：SQLite 数据库路径
- `report_dir`：Markdown / HTML 报告输出目录
- `export_dir`：JSON 导出目录
- `poll_minutes`：持续模式下的轮询间隔
- `timezone`：报告日期和数据库时间戳使用的时区

### 6.2 sources 段

- `arxiv.enabled`：是否启用 arXiv
- `arxiv.max_results`：每条查询拉取的最大条数
- `dblp.max_results`：每条查询拉取的最大条数
- `google_scholar_alerts.enabled`：是否启用 Scholar Alert 邮件解析

### 6.3 enrichment 段

- `enabled`：是否启用全文增强功能
- `pdf_dir`：PDF 本地缓存目录
- `text_dir`：提取后的纯文本目录
- `download_timeout_seconds`：下载 PDF 的超时时间
- `max_papers_per_run`：每轮最多增强多少篇论文
- `redownload_existing`：是否覆盖已下载文件
- `use_pdftotext`：优先使用系统 `pdftotext`
- `excerpt_chars`：写入数据库的全文节选长度
- `process_classifications`：默认增强哪些等级的论文

### 6.4 llm 段

- `enabled`：是否启用 LLM 摘要
- `provider`：支持 `openai_compatible` 和 `openai_responses`
- `base_url`：
  - 若是兼容接口，填写支持 `POST /chat/completions` 的前缀
  - 若是 OpenAI 原生接口，填写 `https://api.openai.com/v1`
- `api_key_env`：API Key 对应的环境变量名
- 不要把真实密钥直接填到 `api_key_env`，这里应写环境变量名，例如 `SAURLAX_API_KEY` 或 `IKUNCODE_API_KEY`
- `model`：模型名
- `max_input_chars`：单次 LLM 调用可承载的正文字符预算
- `max_output_tokens`：输出长度上限
- `temperature`：生成温度
- `store`：仅在 `openai_responses` 下生效，是否让 OpenAI 存储该请求
- `enable_topic_digest`：是否允许在报告里生成主题级聚合摘要
- `topic_digest_entry_limit`：每个主题最多取多少篇论文做 LLM 聚合
- `fulltext_chunk_chars`：完整正文分块时，每块大约多少字符
- `fulltext_chunk_overlap_chars`：相邻分块的重叠长度
- `fulltext_max_chunks`：单篇论文最多切成多少块做 map-reduce
- `llm_variants`：额外的模型配置列表，适合在同一个 API 地址下挂多个模型

当前仓库默认就是双模型模式：

- [config.json](/home/momo/git_ws/search/config/config.json)
  - `deepseek-chat`
- [config.example.json](/home/momo/git_ws/search/config/config.example.json)
  - `gpt-5.4`

如果你希望把更多模型接进来，直接往 `llm_variants` 里追加即可。

## 8. Google Scholar Alerts 配置方法

本程序不直接爬取 Google Scholar 页面，而是读取官方提醒邮件。

### 8.1 推荐流程

1. 用 Google Scholar 为每条重点 query 建立 `Alert`
2. 把提醒邮件发到一个专用邮箱
3. 在邮箱里开启 `IMAP`
4. 配置 `config/config.json` 里的 `google_scholar_alerts`
5. 设置环境变量中的邮箱密码

### 8.2 配置示例

```json
"google_scholar_alerts": {
  "enabled": true,
  "imap_host": "imap.gmail.com",
  "imap_port": 993,
  "username": "your_mail@example.com",
  "password_env": "SCHOLAR_ALERTS_PASSWORD",
  "folder": "INBOX",
  "subject_keyword": "Google Scholar Alerts",
  "search_criterion": "UNSEEN",
  "timeout_seconds": 20
}
```

然后在 shell 中设置密码：

```bash
export SCHOLAR_ALERTS_PASSWORD='你的邮箱密码或应用专用密码'
```

### 8.3 建议

- 最好使用应用专用密码，不要直接使用主账号密码
- 最好用单独邮箱，不要和日常邮件混在一起
- 默认 `UNSEEN` 会在程序处理完后把邮件标记为已读

## 9. 命令说明

### 9.1 `init`

创建默认配置并初始化目录：

```bash
python3 main.py init
```

如果配置已经存在并希望覆盖：

```bash
python3 main.py init --force
```

### 9.2 `fetch`

只抓取，不生成报告：

```bash
python3 main.py fetch --source arxiv
python3 main.py fetch --source dblp
python3 main.py fetch --source google_scholar_alerts
python3 main.py fetch --source arxiv --source dblp --since-last-run
```

如果不传 `--source`，会对配置中启用的全部数据源抓取。

### 9.3 `report`

只根据数据库内容生成报告：

```bash
python3 main.py report --type daily
python3 main.py report --type weekly --days 7
python3 main.py report --date 2026-03-10 --type daily
python3 main.py report --type daily --with-llm
```

### 9.4 `enrich`

下载 PDF、提取全文，并根据完整正文刷新摘要：

```bash
python3 main.py enrich
python3 main.py enrich --limit 10
python3 main.py enrich --topic ai_operator_acceleration
python3 main.py enrich --classification relevant
python3 main.py enrich --with-llm
python3 main.py enrich --with-llm --since-last-run
```

说明：

- `--limit` 控制本轮最多增强多少篇
- `--topic` 只增强指定主题
- `--classification` 只增强 `relevant` 或 `maybe`
- `--force` 会忽略本地缓存重新处理
- `--with-llm` 会在已配置 LLM 的前提下强制尝试生成 LLM 摘要
- `--since-last-run` 会只增强自上次成功增强后新增入库的论文
- 不加 `--skip-pdf` 时，LLM 会优先读取完整 `fulltext_txt_path`，按分块分析整篇论文，再聚合成最终总结

### 9.5 `run-once`

抓取 + 生成报告：

```bash
python3 main.py run-once --type daily
python3 main.py run-once --type weekly --days 7
python3 main.py run-once --type daily --since-last-run
```

抓取 + 全文增强 + 报告：

```bash
python3 main.py run-once --type daily --enrich
python3 main.py run-once --type daily --enrich --enrich-limit 10
python3 main.py run-once --type daily --enrich --with-llm
python3 main.py run-once --type daily --since-last-run --enrich --with-llm
```

### 9.6 `daemon`

持续轮询抓取，并每轮覆盖更新当天报告：

```bash
python3 main.py daemon --type daily
python3 main.py daemon --type daily --since-last-run
```

每轮抓取后自动做全文增强：

```bash
python3 main.py daemon --type daily --enrich
python3 main.py daemon --type daily --enrich --with-llm
python3 main.py daemon --type daily --since-last-run --enrich --with-llm
```

测试时只跑一轮：

```bash
python3 main.py daemon --type daily --loops 1
```

## 11. 评分与摘要机制

### 11.1 评分逻辑

程序会把标题、摘要、venue、分类拼在一起打分：

- 命中 `must_match_groups` 会大幅加分
- 命中 `positive_keywords` 会继续加分
- 命中 `exclude_keywords` 会扣分
- 命中 `arxiv_categories` 和 `dblp_venue_keywords` 会小幅加分

最终分为三类：

- `relevant`
- `maybe`
- `irrelevant`

报告默认包含 `relevant + maybe`。

### 11.2 机器摘要

当前版本有两种摘要模式：

1. 离线摘要
2. LLM 摘要

离线摘要不依赖外部服务：

- 优先提取摘要前 1 到 2 句
- 如果已抽取全文，LLM 会优先读取完整 `fulltext_txt_path`，而不是只读数据库里的 `fulltext_excerpt`
- 拼上命中的关键词和主题标签
- 明确标注总结依据是 `metadata`、`abstract` 还是 `fulltext`

这样做的好处是：

- 可直接离线运行
- 不需要额外 API Key
- 成本为 0

如果你启用了 `llm.enabled=true` 且配置了兼容接口，程序会在 `enrich --with-llm` 或配置默认启用时生成更强的中文结构化摘要，包含：

- `summary`
- `problem`
- `method`
- `contributions`
- `limitations`
- `tags`
- `basis`

如果你同时开启了 `llm.enable_topic_digest=true`，并在生成报告时使用：

```bash
python3 main.py report --type daily --with-llm
```

则每个主题上方还会多出一段聚合摘要，包含：

- `LLM 主题概览`
- `LLM 关键观察`
- `LLM 建议关注`
- `LLM Token` 统计

### 11.3 LLM 配置示例

如果你要接兼容 `chat/completions` 的服务，可在配置中填写：

```json
"llm": {
  "enabled": true,
  "provider": "openai_compatible",
  "base_url": "https://your-endpoint.example/v1",
  "api_key_env": "LLM_API_KEY",
  "model": "your-model-name",
  "timeout_seconds": 60,
  "temperature": 0.2,
  "max_input_chars": 16000,
  "max_output_tokens": 700,
  "store": false,
  "enable_topic_digest": true,
  "topic_digest_entry_limit": 8
}
```

然后在 shell 中设置：

```bash
export LLM_API_KEY='你的密钥'
```

如果 `enabled=false`，程序会自动退回离线摘要。

### 11.4 OpenAI 原生接口配置示例

如果你要直接接 OpenAI API，并保持当前程序的通用性，推荐把 provider 切到 `openai_responses`：

```json
"llm": {
  "enabled": true,
  "provider": "openai_responses",
  "base_url": "https://api.openai.com/v1",
  "api_key_env": "OPENAI_API_KEY",
  "model": "gpt-5-mini",
  "timeout_seconds": 60,
  "temperature": 0.2,
  "max_input_chars": 16000,
  "max_output_tokens": 700,
  "store": false,
  "enable_topic_digest": true,
  "topic_digest_entry_limit": 8
}
```

然后设置：

```bash
export OPENAI_API_KEY='你的 OpenAI API Key'
```

说明：

- `openai_compatible` 继续保留，适合任何兼容 `chat/completions` 的服务
- `openai_responses` 会启用 OpenAI 原生 `Responses API`
- 原生模式下，单篇总结和主题聚合都会走结构化输出，稳定性比纯提示 JSON 更好

## 14. 常见问题

### 14.1 为什么第一次抓到很多历史论文

因为程序默认会按查询拉取当前 API 返回的最近一批结果，再通过数据库去重。第一次运行相当于做一次小规模回填，后续运行就会稳定很多。

### 14.2 为什么 DBLP 结果没有摘要

DBLP 主要提供的是题目、作者、venue、年份等元数据，不保证有摘要，这是正常现象。

### 14.3 为什么 Google Scholar 默认没开

因为它依赖邮箱和 IMAP 配置，而且推荐以 Alert 邮件方式接入，而不是网页抓取。

### 14.4 报告里为什么会出现 `maybe`

`maybe` 表示相关性中等，程序认为它值得人工扫一眼，但不保证强相关。你可以：

- 提高 `threshold`
- 增加 `must_match_groups`
- 增加 `exclude_keywords`

### 14.5 为什么有些论文显示 `PDF pending` 或 `全文状态 empty`

这通常表示你还没有运行 `enrich`，或者这篇论文没有可解析的 PDF 链接。先执行：

```bash
python3 main.py enrich --limit 20
```

### 14.6 为什么启用了 LLM 但还是离线摘要

优先检查：

- `llm.enabled` 是否为 `true`
- `base_url` 和 `model` 是否已填写
- `LLM_API_KEY` 是否已设置
- 当前接口是否兼容 `chat/completions`，或是否正确设置为 `openai_responses`

### 14.7 为什么我明明开了 LLM，但总结看起来不像全文总结

优先检查：

- 这篇论文是否已经有 `fulltext_txt_path`
- 你运行 `enrich` 时是否误用了 `--skip-pdf`
- 该 PDF 是否抽取失败，导致只能回退到摘要模式
- 报告里的 `总结来源` 是否显示为 `llm+fulltext+metadata`

## 15. 故障排查

### 15.1 报错 `python: command not found`

请改用：

```bash
python3 main.py --help
```

### 15.2 抓取失败

优先检查：

- 当前机器是否能访问外网
- 公司网络是否拦截 `arXiv` / `DBLP`
- `timeout_seconds` 是否过小

### 15.3 PDF 下载或全文抽取失败

优先检查：

- 当前机器是否能访问论文 PDF 链接
- `artifacts/pdfs/` 是否有写权限
- 系统是否安装了 `pdftotext` 和 `pdfinfo`
- 该 PDF 是否是扫描版或图片版，导致文本难以抽取

### 15.4 Scholar Alerts 无结果

优先检查：

- 是否真的收到了 Alert 邮件
- `IMAP` 是否开启
- `password_env` 指向的环境变量是否已设置
- `subject_keyword` 是否和邮件标题匹配
- 邮件是否已经被提前标记为已读

### 15.5 LLM 调用失败

优先检查：

- `base_url` 是否包含正确的 `/v1` 前缀
- 服务是否支持 `POST /chat/completions`，或在 OpenAI 模式下支持 `POST /responses`
- `api_key_env` 指向的环境变量是否存在
- 代理、防火墙或公司网络是否拦截了 LLM 服务
