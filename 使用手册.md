# 论文监控程序使用手册

## 1. 项目简介

本程序用于本地监控你关心的论文主题，当前默认已经配置好两个方向：

- `有限元分析 Matrix-Free 算法优化`
- `AI 算子加速`

支持的数据源和处理方式如下：

- `arXiv`：通过官方 API 直接抓取题目、摘要、分类和链接
- `DBLP`：通过官方搜索 API 补会议/期刊元数据
- `Google Scholar Alerts`：不直接爬取 Scholar 页面，而是解析官方 Alert 邮件

程序会把论文统一写入本地 `SQLite` 数据库，再生成：

- `Markdown` 报告
- `HTML` 报告
- `JSON` 导出

## 2. 功能概览

当前版本具备以下能力：

- 多数据源抓取
- 关键词规则打分
- 去重合并
- 自动生成机器摘要
- 本地保存日报 / 周报
- 支持持续轮询模式
- 支持 Google Scholar Alert 邮箱解析

## 3. 环境要求

- `Python 3.11+`
- Linux / macOS / WSL 均可
- 不依赖第三方 Python 库，直接用标准库即可运行

建议在项目根目录执行所有命令。

## 5. 快速开始

### 5.1 直接运行

如果你已经在项目根目录，最简单的方式是：

```bash
python3 main.py report --date 2026-03-10 --type daily
```

这个命令会：

- 自动初始化数据库结构
- 读取 `config/config.json`
- 在 `reports/daily/` 下生成一份日报

如果数据库里还没有论文，它会生成一份空报告，用来验证程序是否可运行。

### 5.2 抓取论文

抓取 `arXiv`：

```bash
python3 main.py fetch --source arxiv
```

抓取 `DBLP`：

```bash
python3 main.py fetch --source dblp
```

抓取后生成日报：

```bash
python3 main.py report --type daily
```

一次完成抓取和报告：

```bash
python3 main.py run-once --type daily
```

### 5.3 可选安装为命令行工具

如果你希望使用 `paper-monitor` 命令：

```bash
python3 -m pip install -e .
paper-monitor --help
```

## 6. 默认配置说明

默认配置文件位于：

```text
config/config.json
```

程序已经为你的两个主题预置了：

- `source_queries`
- `must_match_groups`
- `positive_keywords`
- `exclude_keywords`
- `arxiv_categories`
- `DBLP venue 关键词`

### 6.1 关键字段解释

- `database_path`：SQLite 数据库路径
- `report_dir`：Markdown / HTML 报告输出目录
- `export_dir`：JSON 导出目录
- `poll_minutes`：持续模式下的轮询间隔
- `timezone`：报告日期和数据库时间戳使用的时区

### 6.2 sources 段

- `arxiv.enabled`：是否启用 arXiv
- `arxiv.max_results`：每条查询拉取的最大条数
- `dblp.max_results`：每条查询拉取的最大条数
- `google_scholar_alerts.enabled`：是否启用 Scholar Alert 邮件解析

### 6.3 topic 段

每个主题包含：

- `id`：内部标识
- `display_name`：报告中显示的名称
- `description`：主题说明
- `source_queries`：不同数据源对应的检索语句
- `must_match_groups`：至少要命中的关键词组
- `positive_keywords`：增强打分关键词
- `exclude_keywords`：排除词
- `threshold`：相关性阈值

## 7. 两个默认主题怎么改

### 7.1 有限元分析 Matrix-Free 算法优化

当前重点关注：

- `matrix-free`
- `partial assembly`
- `sum factorization`
- `finite element / FEM`
- `libCEED / MFEM / deal.II`
- `multigrid / preconditioner`

如果你想进一步偏向 GPU 或高阶方法，可以在 `positive_keywords` 里追加：

```json
[
  "cuda",
  "hip",
  "high-order",
  "spectral element",
  "vectorization"
]
```

### 7.2 AI 算子加速

当前重点关注：

- `kernel fusion`
- `operator fusion`
- `tensor compiler`
- `MLIR / Triton / TVM / XLA`
- `FlashAttention`
- `CUDA / GEMM / inference / training`

如果你更关心某个具体算子方向，比如 attention / sparse / quantization，可以直接在：

- `source_queries`
- `positive_keywords`

中补充对应词条。

## 8. Google Scholar Alerts 配置方法

本程序不直接爬取 Google Scholar 页面，而是读取官方提醒邮件。

### 8.1 推荐流程

1. 用 Google Scholar 为每条重点 query 建立 `Alert`
2. 把提醒邮件发到一个专用邮箱
3. 在邮箱里开启 `IMAP`
4. 配置 `config/config.json` 里的 `google_scholar_alerts`
5. 设置环境变量中的邮箱密码

### 8.2 配置示例

```json
"google_scholar_alerts": {
  "enabled": true,
  "imap_host": "imap.gmail.com",
  "imap_port": 993,
  "username": "your_mail@example.com",
  "password_env": "SCHOLAR_ALERTS_PASSWORD",
  "folder": "INBOX",
  "subject_keyword": "Google Scholar Alerts",
  "search_criterion": "UNSEEN",
  "timeout_seconds": 20
}
```

然后在 shell 中设置密码：

```bash
export SCHOLAR_ALERTS_PASSWORD='你的邮箱密码或应用专用密码'
```

### 8.3 建议

- 最好使用应用专用密码，不要直接使用主账号密码
- 最好用单独邮箱，不要和日常邮件混在一起
- 默认 `UNSEEN` 会在程序处理完后把邮件标记为已读

## 9. 命令说明

### 9.1 `init`

创建默认配置并初始化目录：

```bash
python3 main.py init
```

如果配置已经存在并希望覆盖：

```bash
python3 main.py init --force
```

### 9.2 `fetch`

只抓取，不生成报告：

```bash
python3 main.py fetch --source arxiv
python3 main.py fetch --source dblp
python3 main.py fetch --source google_scholar_alerts
```

如果不传 `--source`，会对配置中启用的全部数据源抓取。

### 9.3 `report`

只根据数据库内容生成报告：

```bash
python3 main.py report --type daily
python3 main.py report --type weekly --days 7
python3 main.py report --date 2026-03-10 --type daily
```

### 9.4 `run-once`

抓取 + 生成报告：

```bash
python3 main.py run-once --type daily
python3 main.py run-once --type weekly --days 7
```

### 9.5 `daemon`

持续轮询抓取，并每轮覆盖更新当天报告：

```bash
python3 main.py daemon --type daily
```

测试时只跑一轮：

```bash
python3 main.py daemon --type daily --loops 1
```

## 11. 评分与摘要机制

### 11.1 评分逻辑

程序会把标题、摘要、venue、分类拼在一起打分：

- 命中 `must_match_groups` 会大幅加分
- 命中 `positive_keywords` 会继续加分
- 命中 `exclude_keywords` 会扣分
- 命中 `arxiv_categories` 和 `dblp_venue_keywords` 会小幅加分

最终分为三类：

- `relevant`
- `maybe`
- `irrelevant`

报告默认包含 `relevant + maybe`。

### 11.2 机器摘要

当前版本的摘要是规则生成的，不依赖外部 LLM：

- 优先提取摘要前 1 到 2 句
- 拼上命中的关键词和主题标签
- 明确标注总结仅基于 `title / abstract / metadata`

这样做的好处是：

- 可直接离线运行
- 不需要额外 API Key
- 成本为 0

## 14. 常见问题

### 14.1 为什么第一次抓到很多历史论文

因为程序默认会按查询拉取当前 API 返回的最近一批结果，再通过数据库去重。第一次运行相当于做一次小规模回填，后续运行就会稳定很多。

### 14.2 为什么 DBLP 结果没有摘要

DBLP 主要提供的是题目、作者、venue、年份等元数据，不保证有摘要，这是正常现象。

### 14.3 为什么 Google Scholar 默认没开

因为它依赖邮箱和 IMAP 配置，而且推荐以 Alert 邮件方式接入，而不是网页抓取。

### 14.4 报告里为什么会出现 `maybe`

`maybe` 表示相关性中等，程序认为它值得人工扫一眼，但不保证强相关。你可以：

- 提高 `threshold`
- 增加 `must_match_groups`
- 增加 `exclude_keywords`

## 15. 故障排查

### 15.1 报错 `python: command not found`

请改用：

```bash
python3 main.py --help
```

### 15.2 抓取失败

优先检查：

- 当前机器是否能访问外网
- 公司网络是否拦截 `arXiv` / `DBLP`
- `timeout_seconds` 是否过小

### 15.3 Scholar Alerts 无结果

优先检查：

- 是否真的收到了 Alert 邮件
- `IMAP` 是否开启
- `password_env` 指向的环境变量是否已设置
- `subject_keyword` 是否和邮件标题匹配
- 邮件是否已经被提前标记为已读
