# 开发者手册

## 1. 项目目标

这是一个本地运行的论文监控工具，当前围绕两个方向工作：

- `matrix_free_fem`
- `ai_operator_acceleration`

项目的主链路是：

1. 抓取论文元数据
2. 去重并写入数据库
3. 对每篇论文做主题匹配与打分
4. 下载 PDF，抽取全文
5. 调用一个或多个 LLM 生成逐篇总结
6. 基于逐篇总结生成日报、周报、论文库总览

## 3. 配置体系

当前常用配置：

- `config/config-poe.json`
  - 主配置
  - 当前主模型：`gpt-5.4`
- `config/config-ikun.json`
  - 第二模型
  - 当前模型：`gpt-5.4`
- `config/config-SAURLAX.json`
  - 可选旧配置
- `config/config.example.json`
  - 纯模板

关键规则：

- `--config` 决定：
  - 抓取关键词
  - 评分规则
  - 数据库路径
  - 报告路径
  - 默认总结来源
- `--extra-config` 只增加额外 LLM
  - 不参与抓取
  - 不影响关键词

多模型模式的硬约束在 `paper_monitor/llm_registry.py`：

- 所有配置必须指向同一个数据库
- 所有配置必须使用同一组 `topic id`
- `variant_id` 不能重复

## 4. 数据流

### 4.1 `fetch`

入口：

- `paper_monitor/pipeline.py`

流程：

1. 读取配置中的 `topics[].source_queries`
2. 调用 `arxiv / dblp / scholar alerts` 抓取
3. 生成 `PaperCandidate`
4. 用 `doi / arxiv_id / title_norm` 去重
5. 写入 `papers` 和 `paper_sources`
6. 对每个 topic 做打分，写入 `matches`
7. 如果当前不是 LLM 总结结果，则生成默认总结

### 4.2 `enrich`

入口：

- `paper_monitor/enrichment.py`

流程：

1. 取出需要增强的论文
2. 下载 PDF
3. 抽全文文本
4. 根据当前启用模型生成逐篇总结
5. 把逐篇总结写入 `paper_llm_summaries`
6. 把主模型结果同步写回 `papers.summary_text`

说明：

- `papers.summary_text` 是“默认总结”
- `paper_llm_summaries` 是“每个模型的独立总结”
- 默认总结永远只会指向“本次运行的第一个模型”
- 当前推荐工作流是两阶段：
  - `poe` 处理全部新增论文，做首轮 PDF 总结和筛选
  - `ikun` 只处理高优先级论文，做第二层深读
- 两阶段入口参数在 `enrich / run-once / daemon`：
  - `--secondary-on-priority`
  - `--secondary-top-per-topic`
  - `--secondary-min-score`
- 如果用户明确指定 `--config config/config-ikun.json` 或 `--paper-id`，仍然可以单独让 `ikun` 深读某篇论文

### 4.3 `report / catalog-report / paper-report`

入口：

- `paper_monitor/reports.py`

规则：

- 报告默认只显示“本次命令启用的模型”
- 历史上数据库里残留的其他模型摘要不会自动混入显示
- 领域聚合总结优先使用“该模型自己生成的逐篇总结”
- 缺失时才回退到默认总结

## 5. 数据库设计

数据库是 SQLite，路径通常是：

- `data/papers.db`

核心表：

- `papers`
  - 论文主表
  - 包含元数据、默认总结、PDF 状态、全文状态
- `paper_sources`
  - 来源表
  - 记录这篇论文是从哪个源、哪个查询抓到的
- `matches`
  - 主题匹配结果
  - 一篇论文可命中多个 topic
- `paper_llm_summaries`
  - 每个模型的逐篇总结
- `checkpoints`
  - 记录 `since-last-run` 的抓取 / 增强进度
- `reports`
  - 记录日报 / 周报路径

去重逻辑：

1. 优先 `doi`
2. 再看 `arxiv_id`
3. 最后看 `title_norm`

## 6. LLM 路由逻辑

关键实现：

- `paper_monitor/llm.py`

单篇总结顺序：

1. 如果本地已有 PDF，优先尝试直接 PDF 输入
2. 对 `chat/completions + PDF` 这类后端，可能先走一轮轻量 `pdf_brief`
3. 如果直接 PDF 失败，再回退到“全文文本分块总结”
4. 如果全文也不可用，再退到“摘要 + 元数据”

当前代码里已经支持的 PDF 路由标签包括：

- `chat_file`
- `chat_input_file`

当前与后端相关的经验约定：

- `poe / gpt-5.4`
  - 更适合做全量首轮筛选
  - `pdf_brief` 推荐 `high`
- `ikun / gpt-5.4`
  - 更适合做重点论文深读
  - `pdf_brief` 推荐 `xhigh`

领域聚合总结不是再读一遍 PDF，而是读取：

- 当前模型自己的逐篇总结
- 或默认总结

然后再生成该领域的聚合结论。

## 7. 提示词结构

提示词文件都在：

- `prompts/`

主要分三类：

- 单篇总结
  - `paper_summary_system.txt`
  - `paper_summary_user.txt`
- 全文分块 map-reduce
  - `paper_chunk_system.txt`
  - `paper_chunk_user.txt`
  - `paper_reduce_system.txt`
  - `paper_reduce_user.txt`
- 领域聚合总结
  - `topic_digest_system.txt`
  - `topic_digest_user.txt`

如果要改“单篇输出结构”，优先看单篇提示词。  
如果要改“整篇论文怎么分块、怎么聚合”，优先看 map-reduce 提示词。  
如果要改“日报 / 周报里的领域总结风格”，看 topic digest 提示词。

## 9. 当前默认行为与约定

### 9.1 当前主用模型

- 默认 `--config` 是 `config/config-poe.json`
- 常见双模型组合是：
  - `--config config/config-poe.json`
  - `--extra-config config/config-ikun.json`

### 9.2 当前主题

- `matrix_free_fem`
  - 偏实现与性能优化
  - 不偏纯数学分析
- `ai_operator_acceleration`
  - 包含 kernel fusion、compiler、GEMM、Tensor Core、ARM SVE/SME

### 9.3 当前显示逻辑

报告只显示当前命令启用的模型。  
这是为了防止数据库里历史模型摘要把当前输出污染掉。

## 10. 常用开发命令

### 10.1 跑测试

```bash
python3 -m unittest discover -s tests -v
```

### 10.2 初始化空库

```bash
python3 main.py init --force
```

### 10.3 首次抓取

```bash
python3 main.py --config config/config-poe.json fetch --source arxiv --source dblp --recent-limit 100 --page-size 50
```

### 10.4 首次双模型增强

```bash
python3 main.py --config config/config-poe.json enrich --limit 200 --with-llm --extra-config config/config-ikun.json --workers 1 --secondary-on-priority --secondary-top-per-topic 5 --secondary-min-score 26
```

### 10.5 生成总览

```bash
python3 main.py --config config/config-poe.json catalog-report --extra-config config/config-ikun.json
```

### 10.6 查找 paper_id

```bash
python3 main.py --config config/config-poe.json paper-find --title "Hexcute"
python3 main.py --config config/config-poe.json paper-find --doi "10.1000/manual-study"
python3 main.py --config config/config-poe.json paper-find --arxiv-id "2504.16214v3"
```

### 10.7 手动让 Ikun 深读指定论文

```bash
python3 main.py --config config/config-ikun.json enrich --paper-id 123 --with-llm --workers 1 --force
```

### 10.8 每日增量

```bash
python3 main.py --config config/config-poe.json run-once --type daily --source arxiv --source dblp --page-size 20 --since-last-run --enrich --enrich-limit 20 --with-llm --extra-config config/config-ikun.json --workers 1 --secondary-on-priority --secondary-top-per-topic 5 --secondary-min-score 26
```

## 11. 常见改动点

### 11.1 改抓取关键词

改主配置文件里的：

- `topics[].source_queries`
- `topics[].must_match_groups`
- `topics[].positive_keywords`
- `topics[].exclude_keywords`
- `topics[].priority_arxiv_categories`
- `topics[].priority_venue_keywords`

### 11.2 增加新 LLM

通常不需要改代码，只需要：

1. 复制一份配置
2. 改 `llm.variant_id / label / provider / base_url / api_key_env / model`
3. 用 `--extra-config` 接入

### 11.3 调整 PDF 总结策略

重点看：

- `paper_monitor/llm.py`
- `LLMConfig.pdf_input_mode`
- `LLMConfig.pdf_inline_max_bytes`
- `LLMConfig.reasoning_by_task`
- `LLMConfig.model_reasoning_effort`

如果是 `chat/completions + PDF` 后端，还要关注：

- `reasoning_by_task.pdf_brief`
- PDF 内容块顺序和结构化输出约束是否与该网关兼容

### 11.4 改总览 / 日报 / 周报结构

重点看：

- `paper_monitor/reports.py`

### 11.5 改默认总结逻辑

重点看：

- `paper_monitor/summarize.py`
- `paper_monitor/enrichment.py`

## 12. 已知实现约束

### 12.1 PDF 直读能力依赖后端

不同兼容网关的行为不一致。  
所以代码不能假设某个模型永远支持 PDF，必须走探测逻辑。

### 12.2 `since-last-run` 依赖 checkpoint

抓取和增强各自维护 checkpoint。  
如果手动清库但不清 checkpoint，语义可能和预期不一致。

### 12.3 默认总结和多模型总结不是同一层数据

- `papers.summary_text` 是单值
- `paper_llm_summaries` 是多值

很多报表 bug 都来自把这两层混为一谈。

### 12.4 DBLP 会偶发 5xx

当前逻辑已经做了重试和降级，但开发时不要把 DBLP 视为稳定强源。

## 13. 推荐的开发顺序

如果后面继续开发，建议按这个顺序思考：

1. 先确认改动发生在哪一层：
   - 抓取
   - 评分
   - 全文增强
   - LLM 路由
   - 报告渲染
2. 再确认是否会影响数据库结构
3. 再补最小测试
4. 最后再更新用户手册

## 14. 后续适合继续做的方向

- 引用链扩展源
  - 例如 OpenAlex / Semantic Scholar
- GitHub 辅助发现源
  - 例如 CUTLASS / Triton / FlashAttention
- 更强的 PDF / 版式解析器
  - 例如 GROBID / Marker
- 更稳定的成本审计
  - 记录每个模型的 token 与费用
- 后台自动调度
  - systemd timer / cron
