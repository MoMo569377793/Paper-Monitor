# 开发者手册

## 1. 项目定位

这是一个本地运行的论文监控系统，围绕两个主题工作：

- `matrix_free_fem`
- `ai_operator_acceleration`

系统目标不是“做一个通用搜索引擎”，而是做一条稳定的研究情报流水线：

1. 抓论文
2. 去重和打分
3. 解析和下载 PDF / 抽全文
4. 生成单篇多模型总结
5. 生成领域级聚合报告
6. 支持手动维护论文库

如果只是想快速找命令，不看实现细节，可以先看：

- `命令速查表.md`

## 2. 当前推荐运行策略

当前代码和文档假设的推荐策略是：

- 首次全量 `100 + 100` 篇论文：
  - 单篇总结优先让 `Ikun / gpt-5.4` 负责
- 后续增量论文：
  - 单篇总结可以 `Poe + Ikun` 同时生成
- 日报 / 周报 / 总览里的领域级聚合总结：
  - 优先让主模型 `Poe / Claude` 负责
- 当新增论文很多时：
  - `Poe` 处理全部
  - `Ikun` 只处理高优先级论文

这套策略是通过两组机制实现的：

- `secondary_on_priority`
  - 控制额外模型只处理高优先级论文
- `digest_primary_only`
  - 控制领域级聚合只由主模型生成

性能层面还有一条新规则：

- `enrich --workers N`
  - 现在对完整增强主流程生效
  - 会并发执行 `下载 PDF -> 抽全文 -> LLM 单篇总结`
  - 不再只对 `--skip-pdf` 的 LLM-only 路径生效

### 5.2 `enrich`

CLI：

- `paper_monitor/cli.py`

主逻辑：

- `EnrichmentPipeline.run()`

涉及：

- `DocumentProcessor.enrich()`
- `LLMClient.generate_summary()`
- `Database.upsert_paper_llm_summary()`
- `Database.update_paper_analysis()`

当前 `enrich` 还有两条重要恢复策略：

- 已有完整 `pdf_direct` 结果的 `(paper_id, variant_id)` 会被跳过
- 之前只得到 fallback 结果的 `(paper_id, variant_id)` 会自动重试
- 现在还支持跨模型补跑：
  - `--retry-from-variant ikun --retry-from-status incomplete`
  - 表示“当前模型只处理 `Ikun` 仍缺失或回退的论文”

### 5.3 当前真实的数据流

现在的主流程不是“抓到论文以后立刻一定有 PDF”，而是：

1. `fetch`
   - 从 `arXiv` / `DBLP` 取元数据
   - 统一去重、打分、入库
2. `enrich`
   - 为库里的论文解析 PDF 地址
   - 下载 PDF
   - 抽取全文文本
   - 跑单篇 LLM 总结
3. `report`
   - 读取数据库中的论文、逐篇总结、领域聚合结果，生成输出

#### 5.3.3 空窗期报告回退策略

为了避免严格规则下日报 / 周报完全空白，`generate_report()` 现在有一层报告级回退：

- 先按正常窗口读取严格命中论文
- 如果某个 topic 在该窗口内没有严格命中论文
- 则自动补一个“高分回顾”窗口

当前规则：

- `matrix_free_fem`
  - 回顾近 `90` 天高分论文
- `ai_operator_acceleration`
  - 回顾近 `30` 天高分论文

注意：

- 这不会影响抓取、打分和数据库里的 `matches`
- 只是报告层的展示补充
- 领域级 `topic_digest` 在严格窗口为空时，也会改用这批回顾条目生成
- JSON 导出里会额外写入 `fallback_reviews`

#### 5.3.1 `DocumentProcessor.resolve_pdf_url()` 的当前顺序

当前 PDF 解析顺序是：

1. 原始元数据里的 `pdf_url`
2. 从 `DOI / arXiv / CoRR` 线索反推出 `arXiv PDF`
3. `arXiv abs -> pdf` 改写
4. landing page 抓取
   - `citation_pdf_url`
   - `dc.identifier`
   - `og:url`
   - 任意 `.pdf` 链接
5. `Semantic Scholar openAccessPdf`
   - 优先 `DOI` / `ARXIV` details 接口
   - 如果设置了 `SEMANTIC_SCHOLAR_API_KEY`，再补做标题搜索
6. 最后才尝试把 `primary_url` 当 PDF 候选地址

实现入口在：

- `paper_monitor/enrichment.py`
  - `DocumentProcessor.resolve_pdf_url()`
  - `_infer_arxiv_id()`
  - `_infer_pdf_url_from_landing_page()`
  - `_infer_pdf_url_from_semantic_scholar()`

#### 5.3.2 `Semantic Scholar` 的定位

它现在是“补 PDF 发现源”，不是主论文源。

为什么要这样设计：

- 主论文抓取仍然靠 `arXiv + DBLP`
- 但很多 DOI 论文并没有直接暴露 PDF 地址
- `Semantic Scholar details` 有时能给出 `openAccessPdf`
- 尤其适合：
  - DOI 页面被拦
  - landing page 没有直接 PDF
  - 但论文其实存在开放获取副本

当前实现细节：

- 默认不要求 API key
- 无 key 时仍会尝试 details 接口
- 有 `SEMANTIC_SCHOLAR_API_KEY` 时，会额外开启标题搜索
- details/search 遇到 `429` 会做短退避重试

#### 5.3.3 目前已验证的 live 结果

这不是纯理论设计，当前代码已经用真实论文验证过：

- landing page 补回成功：
  - `paper 76`
  - `paper 188`
- `Semantic Scholar openAccessPdf` 补回成功：
  - `paper 89`
- `Semantic Scholar` 虽返回 PDF 地址，但源站仍会拦：
  - `paper 77`
  - ACM PDF 返回 `403`
- landing page 给出 PDF 候选，但源站返回 HTML：
  - `paper 90`
  - Springer `citation_pdf_url` 实际返回 HTML

这说明：

- `Semantic Scholar` 很有用
- 但仍受源站权限/反爬限制
- 所以 `no-pdf` 状态仍然会保留一小部分论文

#### 5.3.4 下载、抽全文与回退顺序

一旦 `resolve_pdf_url()` 拿到候选 URL，`DocumentProcessor.enrich()` 会：

1. 下载 PDF
2. 验证返回体是否真的是 PDF
3. 抽全文文本
4. 更新 `papers.pdf_*` 和 `papers.fulltext_*`

如果下载 URL 最终返回：

- `403`
- HTML 页面
- 非 PDF 内容

现在都会被安全地归成：

- `pdf_status = no-pdf`
- `fulltext_status = empty`

不会再因为单个被拦的 `openAccessPdf` 链接把整个文档刷新脚本打崩。

- `variant_id` 不能重复

## 7. 数据流

### 7.1 `fetch` 数据流

1. 读取 `topics[].source_queries`
2. 调用抓取器获取 `PaperCandidate`
3. 通过 `Semantic Scholar`（优先 `DOI/arXiv`，再标题）补一层引用元数据
4. 按 topic 先做硬性条件过滤，再做关键词/引用/时间综合排序
5. 在 `recent_limit` 窗口内混合保留“高引用经典论文”和“近期论文”
6. 注入 `topics[].seed_papers` 里的预置种子论文
7. 用 `doi / arxiv_id / title_norm` 去重
8. 写入 `papers` 和 `paper_sources`
9. 计算 `matches`
10. 如果没有 LLM 总结，则生成离线默认总结

当前默认策略里，`recent_limit=100` 不再代表“最新 100 篇”，而是“尽量保留约一半高引用经典论文，再补齐近期论文”。经典论文的上限在代码里封顶为 `50`。

### 7.2 `enrich` 数据流

1. 读取候选论文
2. 下载 PDF
3. 抽全文文本
4. 按目标模型集合生成逐篇总结
5. 写入 `paper_llm_summaries`
6. 把主模型结果同步到 `papers.summary_text`

### 7.3 `report` 数据流

1. 从时间窗口内读取 `ReportEntry`
2. 读取对应论文的 `paper_llm_summaries`
3. 选择要展示的模型摘要
4. 生成领域级聚合总结
5. 导出单篇报告
6. 渲染 Markdown / HTML / JSON

## 8. 数据库设计

数据库默认路径：

- `data/papers.db`

### 8.1 核心表

- `papers`
  - 主表
  - 存元数据、默认总结、PDF / 全文状态
- `paper_sources`
  - 来源表
  - 记录论文从哪个源、哪个查询抓到
- `matches`
  - 主题匹配结果
- `paper_llm_summaries`
  - 每个模型的逐篇总结
- `checkpoints`
  - `since-last-run` 的抓取 / 增强进度
- `reports`
  - 报告路径记录

### 8.2 去重优先级

1. `doi`
2. `arxiv_id`
3. `title_norm`

### 8.3 两层摘要数据

这是开发时最容易搞混的地方：

- `papers.summary_text`
  - 单值
  - 代表“默认总结”
- `paper_llm_summaries`
  - 多值
  - 每个模型各自一条

很多报表和回归问题都来自把这两层混为一谈。

## 9. LLM 路由逻辑

核心实现：

- `paper_monitor/llm.py`

### 9.1 单篇总结路由

顺序：

1. 如果本地有 PDF，优先尝试直接 PDF 输入
2. 对 `Ikun / gpt-5.4` 这类非 Poe 的 `chat/completions + PDF` 后端，先尝试“结构化直读 PDF”
3. 如果结构化直读失败，再尝试 `pdf_brief`
4. 如果直接 PDF 失败，再回退到全文文本 map-reduce
5. 如果全文也不可用，再退到摘要 + 元数据

### 9.2 当前支持的 PDF 路由标签

- `chat_file`
- `chat_input_file`
- `responses_input_file`

对于 `chat/completions` 路由，当前还做了一个兼容补救：

- 如果首选策略返回
  - `Unknown parameter ... file`
  - 或类似的文件字段不兼容错误
- 代码会自动切到另一种 PDF 内容块格式再试一次

### 9.3 当前后端经验

- `poe / claude-opus-4.6`
  - 适合做领域聚合、日报、周报、总览
  - 也适合作为后续增量论文的主模型
  - `pdf_brief` 建议 `max`
  - 当前 `pdf_input_mode = force`
- `ikun / gpt-5.4`
  - 适合单篇深读
  - 适合首次全量逐篇总结
  - `paper_summary` / `paper_reduce` 建议 `xhigh`
  - 当前 `pdf_input_mode = force`
  - 当前实现会优先尝试 `PDF -> structured JSON prompt`
  - 如果失败，再走 `PDF -> pdf_brief`

### 9.4 Poe / Ikun / Gemini 的参数映射

当前代码里有 3 条主要映射：

- Poe + Claude
  - 使用 `extra_body.output_effort`
- Poe + Gemini
  - 使用 `extra_body.thinking_level`
- Ikun / OpenAI-compatible / 非 Poe Claude / 非 Poe Gemini
  - 使用 `reasoning_effort`

对应代码集中在：

- `LLMClient._apply_chat_request_options()`

## 10. 多模型协作逻辑

### 10.1 `secondary_on_priority`

实现位置：

- `EnrichmentPipeline._build_target_variants_map()`

逻辑：

- 主模型始终处理全部候选论文
- 额外模型只处理高优先级论文

高优先级论文来自：

- `score >= secondary_min_score`
- 每个 topic 的 Top N

### 10.2 `digest_primary_only`

实现位置：

- `generate_report()`
- `generate_catalog_report()`
- `run_daemon()`
- CLI 参数透传

逻辑：

- 单篇多模型摘要仍然全部保留
- 领域级聚合总结只使用主模型

这是当前“后续增量双模型单篇对比，但领域聚合只让 Poe 写”的关键机制。

### 11.5 `topic_digest_*`

用途：

- 日报 / 周报 / 总览里的领域级聚合总结

当前风格：

## 12. 报告系统

### 12.1 报告类型

- `report`

  - 全库总览
- `paper-report`
  - 单篇报告
- `compare-report`
  - 并排模型对比

### 12.2 显示规则

- 报告会优先显示本次命令启用的模型
- 如果同一篇论文数据库里还保存了其他模型结果，也会一并显示
- 领域聚合优先用该模型自己的逐篇总结
- 该模型没有逐篇总结时，才回退到默认总结

这条规则是为了支持：

- 先用 `Ikun` 全量深读
- 后来再用 `Poe` 对同一批或部分论文补跑
- 报表里把两边结果一起展示

- `pdf_inline_max_bytes`
  - 控制允许直接内联上传给模型的 PDF 最大大小
  - 不是越大越好
  - 因为发送时会先转成 base64，实际请求体比原始 PDF 更大
  - 值过大更容易撞到网关体积限制或上传超时

## 17. 已知约束

### 17.1 兼容网关行为不一致

不同兼容网关对：

- PDF 文件块格式
- 结构化输出
- 额外字段

的支持程度不同。

### 17.2 `fetch` 不会下载 PDF

只有 `enrich` 才会真正下载 PDF / 抽全文 / 跑 LLM。

### 17.3 清数据库不等于清缓存

删除 `data/papers.db` 不会自动删除：

- `artifacts/pdfs`
- `artifacts/text`
- `reports`
- `exports`
