# 开发者手册

## 1. 项目定位

这是一个本地运行的论文监控系统，围绕两个主题工作：

- `matrix_free_fem`
- `ai_operator_acceleration`

系统目标不是“做一个通用搜索引擎”，而是做一条稳定的研究情报流水线：

1. 抓论文
2. 去重和打分
3. 下载 PDF / 抽全文
4. 生成单篇多模型总结
5. 生成领域级聚合报告
6. 支持手动维护论文库

如果只是想快速找命令，不看实现细节，可以先看：

- `命令速查表.md`

## 2. 当前推荐运行策略

当前代码和文档假设的推荐策略是：

- 首次全量 `100 + 100` 篇论文：
  - 单篇总结优先让 `Ikun / gpt-5.4` 负责
- 后续增量论文：
  - 单篇总结可以 `Poe + Ikun` 同时生成
- 日报 / 周报 / 总览里的领域级聚合总结：
  - 优先让主模型 `Poe / Claude` 负责
- 当新增论文很多时：
  - `Poe` 处理全部
  - `Ikun` 只处理高优先级论文

这套策略是通过两组机制实现的：

- `secondary_on_priority`
  - 控制额外模型只处理高优先级论文
- `digest_primary_only`
  - 控制领域级聚合只由主模型生成

性能层面还有一条新规则：

- `enrich --workers N`
  - 现在对完整增强主流程生效
  - 会并发执行 `下载 PDF -> 抽全文 -> LLM 单篇总结`
  - 不再只对 `--skip-pdf` 的 LLM-only 路径生效

## 6. 配置体系

### 6.1 当前常用配置

- `config/config-poe.json`
  - 主配置
  - 当前主模型：`claude-opus-4.6`
  - 当前配置特征：
    - `pdf_input_mode = force`
    - `model_output_effort = max`
    - `output_effort_by_task.*`
- `config/config-ikun.json`
  - 第二模型
  - 当前模型：`gpt-5.4`
  - 当前配置特征：
    - `pdf_input_mode = force`
    - `model_reasoning_effort = xhigh`
    - `reasoning_by_task.*`
- `config/config-SAURLAX.json`
  - 可选旧配置
- `config/config.example.json`
  - 模板

### 6.2 `--config` 和 `--extra-config`

这是项目里最重要的一条规则：

- `--config`
  - 决定抓取关键词、打分规则、数据库路径、报告路径、默认总结来源
- `--extra-config`
  - 只增加额外模型
  - 不参与抓取，不影响关键词

### 6.3 多模型运行时约束

在 `paper_monitor/llm_registry.py` 中强制保证：

- 所有配置必须指向同一个数据库
- 所有配置必须使用同一组 `topic id`
- `variant_id` 不能重复

## 7. 数据流

### 7.1 `fetch` 数据流

1. 读取 `topics[].source_queries`
2. 调用抓取器获取 `PaperCandidate`
3. 用 `doi / arxiv_id / title_norm` 去重
4. 写入 `papers` 和 `paper_sources`
5. 计算 `matches`
6. 如果没有 LLM 总结，则生成离线默认总结

### 7.2 `enrich` 数据流

1. 读取候选论文
2. 下载 PDF
3. 抽全文文本
4. 按目标模型集合生成逐篇总结
5. 写入 `paper_llm_summaries`
6. 把主模型结果同步到 `papers.summary_text`

### 7.3 `report` 数据流

1. 从时间窗口内读取 `ReportEntry`
2. 读取对应论文的 `paper_llm_summaries`
3. 选择要展示的模型摘要
4. 生成领域级聚合总结
5. 导出单篇报告
6. 渲染 Markdown / HTML / JSON

## 8. 数据库设计

数据库默认路径：

- `data/papers.db`

### 8.1 核心表

- `papers`
  - 主表
  - 存元数据、默认总结、PDF / 全文状态
- `paper_sources`
  - 来源表
  - 记录论文从哪个源、哪个查询抓到
- `matches`
  - 主题匹配结果
- `paper_llm_summaries`
  - 每个模型的逐篇总结
- `checkpoints`
  - `since-last-run` 的抓取 / 增强进度
- `reports`
  - 报告路径记录

### 8.2 去重优先级

1. `doi`
2. `arxiv_id`
3. `title_norm`

### 8.3 两层摘要数据

这是开发时最容易搞混的地方：

- `papers.summary_text`
  - 单值
  - 代表“默认总结”
- `paper_llm_summaries`
  - 多值
  - 每个模型各自一条

很多报表和回归问题都来自把这两层混为一谈。

## 9. LLM 路由逻辑

核心实现：

- `paper_monitor/llm.py`

### 9.1 单篇总结路由

顺序：

1. 如果本地有 PDF，优先尝试直接 PDF 输入
2. 对 `Ikun / gpt-5.4` 这类非 Poe 的 `chat/completions + PDF` 后端，先尝试“结构化直读 PDF”
3. 如果结构化直读失败，再尝试 `pdf_brief`
4. 如果直接 PDF 失败，再回退到全文文本 map-reduce
5. 如果全文也不可用，再退到摘要 + 元数据

### 9.2 当前支持的 PDF 路由标签

- `chat_file`
- `chat_input_file`
- `responses_input_file`

### 9.3 当前后端经验

- `poe / claude-opus-4.6`
  - 适合做领域聚合、日报、周报、总览
  - 也适合作为后续增量论文的主模型
  - `pdf_brief` 建议 `max`
  - 当前 `pdf_input_mode = force`
- `ikun / gpt-5.4`
  - 适合单篇深读
  - 适合首次全量逐篇总结
  - `paper_summary` / `paper_reduce` 建议 `xhigh`
  - 当前 `pdf_input_mode = force`
  - 当前实现会优先尝试 `PDF -> structured JSON prompt`
  - 如果失败，再走 `PDF -> pdf_brief`

### 9.4 Poe / Ikun / Gemini 的参数映射

当前代码里有 3 条主要映射：

- Poe + Claude
  - 使用 `extra_body.output_effort`
- Poe + Gemini
  - 使用 `extra_body.thinking_level`
- Ikun / OpenAI-compatible / 非 Poe Claude / 非 Poe Gemini
  - 使用 `reasoning_effort`

对应代码集中在：

- `LLMClient._apply_chat_request_options()`

## 10. 多模型协作逻辑

### 10.1 `secondary_on_priority`

实现位置：

- `EnrichmentPipeline._build_target_variants_map()`

逻辑：

- 主模型始终处理全部候选论文
- 额外模型只处理高优先级论文

高优先级论文来自：

- `score >= secondary_min_score`
- 每个 topic 的 Top N

### 10.2 `digest_primary_only`

实现位置：

- `generate_report()`
- `generate_catalog_report()`
- `run_daemon()`
- CLI 参数透传

逻辑：

- 单篇多模型摘要仍然全部保留
- 领域级聚合总结只使用主模型

这是当前“后续增量双模型单篇对比，但领域聚合只让 Poe 写”的关键机制。

## 11. 提示词体系

提示词文件位于：

- `prompts/paper_summary_system.txt`
- `prompts/paper_summary_user.txt`
- `prompts/paper_chunk_system.txt`
- `prompts/paper_chunk_user.txt`
- `prompts/paper_reduce_system.txt`
- `prompts/paper_reduce_user.txt`
- `prompts/topic_digest_system.txt`
- `prompts/topic_digest_user.txt`

### 11.1 `paper_summary_*`

用途：

- 单篇论文最终总结

当前风格：

- 更偏深读
- 更强调优化对象、方法、平台、软件栈、数字结果、局限
- `summary` 比旧版更长

更适合：

- `Ikun` 做单篇深读
- 双模型单篇对比

### 11.2 `pdf_brief`

这不是文件，而是程序内置步骤。

用途：

- 对 `chat/completions + PDF` 后端先生成结构化备忘录

当前风格：

- 固定输出 `摘要 / 问题 / 方法 / 应用 / 结果 / 贡献 / 局限 / 标签`
- 再由程序整理成最终单篇 JSON

更适合：

- `Poe + Claude`

### 11.3 `paper_chunk_*`

用途：

- 全文文本分块分析

### 11.4 `paper_reduce_*`

用途：

- 聚合全文分块分析结果

### 11.5 `topic_digest_*`

用途：

- 日报 / 周报 / 总览里的领域级聚合总结

当前风格：

- `overview` 更长
- `highlights` / `watchlist` 更多
- 更强调“这批论文为什么值得继续跟”

更适合：

- `Poe` 做领域级聚合

## 12. 报告系统

核心文件：

- `paper_monitor/reports.py`

### 12.1 报告类型

- `report`
  - 日报 / 周报
- `catalog-report`
  - 全库总览
- `paper-report`
  - 单篇报告
- `compare-report`
  - 并排模型对比

### 12.2 显示规则

- 报告默认只显示本次命令启用的模型
- 历史上数据库里残留的其他模型摘要不会自动混入显示
- 领域聚合优先用该模型自己的逐篇总结
- 该模型没有逐篇总结时，才回退到默认总结

## 13. 手动论文维护

### 13.1 `paper-add`

手动加入论文并绑定一个或多个 topic。

### 13.2 `paper-find`

当前推荐的查找 `paper_id` 方式。

支持：

- 标题模糊查找
- URL 模糊查找
- DOI 精确查找
- arXiv ID 精确查找
- 按 topic 限定

### 13.3 `paper-delete`

按 `paper_id` 删除论文。

相关记录会级联删除：

- `paper_sources`
- `matches`
- `paper_llm_summaries`

## 17. 已知约束

### 17.1 兼容网关行为不一致

不同兼容网关对：

- PDF 文件块格式
- 结构化输出
- 额外字段

的支持程度不同。

### 17.2 `fetch` 不会下载 PDF

只有 `enrich` 才会真正下载 PDF / 抽全文 / 跑 LLM。

### 17.3 清数据库不等于清缓存

删除 `data/papers.db` 不会自动删除：

- `artifacts/pdfs`
- `artifacts/text`
- `reports`
- `exports`

## 18. 推荐开发顺序

如果后续继续开发，建议按这个顺序判断改动发生在哪一层：

1. 抓取
2. 评分
3. PDF / 全文处理
4. LLM 路由
5. 报告渲染
6. CLI / 调度

先定位层，再改代码。
